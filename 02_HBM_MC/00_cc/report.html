§1What a memory controller actually does

A CPU/GPU core issues simple loads and stores to byte addresses. DRAM does not speak loads/stores — it speaks a small command language over a shared bus, and it has strict timing rules. The memory controller is the translator and scheduler in between.

The DRAM the controller drives

One HBM3 channel here is an array of 32 banks (2 pseudochannels × 4 bankgroups × 4 banks). Each bank is a 2-D grid of 32,768 rows × columns. A bank has one "row buffer": to read any byte you must first copy a whole row into the row buffer, then read columns from it.

CommandMeaningWhen the controller issues it
ACTActivate — open a row into the bank's row bufferbank is closed and you need a row from it
RD / WRRead / Write a column from/to the open row bufferthe wanted row is already open (a "row hit")
PREPrecharge — close the open rowyou need a different row in that bank (a "row conflict")
REFRefresh — recharge leaking cellsperiodically (disabled in this HBM3 config — see §6)

The three outcomes of every access

Row hit
wanted row already open → just RD. Fastest.
Row conflict
wrong row open → PRE → ACT → RD. Slowest.
Row miss
bank closed → ACT → RD. Medium.

Maximizing row hits (and obeying ~20 timing constraints across banks) is the controller's job. Everything below is machinery for that.

§2Ramulator's controller — and why it is "everything in one cycle"

Ramulator 2's GenericDRAMController models that whole pipeline as a single behavioral function called once per DRAM clock. It is wonderfully accurate for DRAM timing, but it deliberately collapses the controller's own pipeline to zero.

Ramulator — generic_dram_controller.cpp tick()
// one call == one DRAM cycle == "the whole pipeline"
void tick() {
  m_clk++;
  serve_completed_reads();         // return finished reads
  m_refresh->tick();               // maybe inject refresh
  auto req = schedule_request();   // pick ONE request:
                                   //   active → priority → read/write
  m_rowpolicy->update(req);         // open-page = no-op
  if (req) {
    update_request_stats(req);     // classify hit/miss/conflict
    issue_command(req);            // drive ONE command to DRAM
  }
}
What real hardware spreads across stages

[look up request queue]
[per-bank FSM: hit? miss? conflict?]
[timing legality check]
[arbitrate: which command this cycle]
[generate DFI command]
[PHY: serialize to DRAM pins]

// Ramulator folds ALL of these into one tick()
The limitation, precisely. Ramulator's controller is ~1 pipeline stage: a single combinational decision over registered state, one command per cycle, on a unified command path. It models none of: the front-end accept pipeline, clock-domain crossing, the DFI sequencer, the PHY, or HBM's separate RAS/CAS command buses. Its JEDEC timing, however, is exact — encoded as per-node "earliest-issuable-clock" registers (§6). That is what we must reproduce.

This is the right tradeoff for a simulator (you want speed and correct DRAM timing, not a gate-level controller). But it means Ramulator cannot answer "how deep is the controller, what does it cost in real cycles/area." Rebuilding it as RTL does — and the RTL becomes a reference design whose behavior is provably the same.

§3Deciding the RTL pipeline (and its depth)

Goal: a synthesizable controller whose issued command stream is identical to Ramulator's. The key design decision is how to map a behavioral tick() onto clocked hardware without changing when each command is issued.

Principle: the decision loop is combinational over registered state

Ramulator's tick() reads registered state (queues, bank states, timing clocks) and produces one command, all "instantly" within a cycle. The RTL mirrors this exactly: the arbitration decision is a combinational cone over registered state, and only the effects (issue, state updates) are clocked. This guarantees the crucial invariant:

ready@T ⇒ issued@T. If a command becomes legal on cycle T, it is issued on cycle T — never one cycle late. Any fixed pipeline delay would shift the command stream and break the bit-exact match.

The send / tick split (the trick that makes it bit-exact)

Ramulator's main loop runs the frontend and the controller at different clock ratios (frontend 8 : memory 3) and always does frontend->tick() (enqueue a request) before memory->tick() (decide). The RTL gives these two actions separate clock phases so a just-enqueued request is visible to the very next decision — and uses cur_clk = clk_cnt + do_tick so the decision sees the post-increment clock, exactly like m_clk++ at the top of tick().

The ~4-stage logical datapath & pipeline depth

S0 · Acceptfront-end gate: enqueue a request if there is room (non-active occupancy ≤ 32)
S1 · Preq + Readyper-entry: needed command (ACT/PRE/RD/WR) + is it legal now (timing)
S2 · ArbitrateFR-FCFS: prefer a ready command, then oldest; active-buffer first
S3 · Issuedrive one command; update bank FSM, timing clocks, buffers, stats
DesignPipeline depth (entry → command)Notes
Ramulator behavioral≈ 1one combinational tick()
This RTL (bit-exact)1 decision cycleS1–S3 combinational; only effects registered, so it matches Ramulator. Read-data return is a separate 11-deep shift pipe (a documented constant, not in the command stream).
Production HBM controller~5–18 (+PHY)S0–S3 would each be registered for timing closure; adds front-end/CDC/DFI/PHY

Why keep the decision combinational rather than pipeline S1–S3? Because the goal is behavioral equivalence to Ramulator, not maximum clock frequency. A real product would register S1→S2→S3 and absorb the resulting constant latency offset; here we keep them in one cycle so the command cycles match Ramulator with zero offset. The microarchitecture is otherwise identical, so the pipelined and bit-exact variants differ only by a known constant.

§6Function-by-function: how each works, in C++ and in RTL

Each block below: why we need it, how Ramulator does it, the equivalent RTL side-by-side, and the difference (if any). Snippets are simplified pseudocode of the real source.

6.1 Address mapping — RoBaRaCoCh

Why: a flat byte address must become (pseudochannel, bankgroup, bank, row, column) so the rest of the controller can track per-bank rows and timing. The bit order chosen (RoBaRaCoCh = Row high, Column low) keeps nearby addresses in the same row → more row hits.

Ramulator — linear_mappers.cpp
addr >>= TX_OFF;        // drop low 4 bits (16B grain)
col = addr & 0x1f;     // 5 bits
pch = (addr>>5) & 1;
bg  = (addr>>6) & 3;
ba  = (addr>>8) & 3;
row = (addr>>10) & 0x7fff;
RTL — addr_mapper.sv (pure comb)
always_comb begin
  a   = addr >> TX_OFF;
  col = a[4:0];
  pch = a[5];
  bg  = a[7:6];
  ba  = a[9:8];
  row = a[24:10];
end

Difference: none — bit-identical. The 4-bit shift is because one DRAM access moves 16 B (prefetch 2 × 8-byte half-bus), so the low 4 address bits never reach the bank.

6.2 Bank FSM & the hit/miss/conflict decision (RequireRowOpen)

Why: before any RD/WR you must know whether the target row is open. This is the single most important decision — it picks the command (ACT, PRE, or RD/WR) and decides the access cost.

Ramulator — RequireRowOpen (lambda)
if (bank.state == Closed)
  preq = ACT;                 // miss
else if (bank.open_row == row)
  preq = final_cmd;           // hit → RD/WR
else
  preq = PRE;                 // conflict → close first
RTL — bank_state.sv / dec()
open = o_open[b];
hit  = open && (o_row[b] == row);
cls  = hit ? HIT : (open ? CONFLICT : MISS);
if      (!open) preq = CMD_ACT;
else if (!hit) preq = CMD_PRE;
else          preq = we ? CMD_WR : CMD_RD;

How it works: ACT sets state=Opened, open_row=row; PRE/PRECHARGE sets state=Closed. Difference: none functionally; the RTL exposes all 32 banks' state as arrays so the arbiter can evaluate every queued request in parallel.

6.3 Timing engine — "earliest-issuable-clock" registers

Why: DRAM has ~20 minimum gaps between commands (e.g. ACT→RD is tRCD=7 cycles; ACT→ACT same bank is tRC=19). The controller may only issue a command when all relevant gaps have elapsed at the bank, bankgroup, pseudochannel and channel levels.

How Ramulator does it: every node keeps m_cmd_ready_clk[cmd] = the earliest clock that command may issue. On each issue, for every timing edge it sets ready_clk[follow] = max(ready_clk[follow], now + latency); check_ready is just clk >= ready_clk[cmd]. The RTL uses the same absolute-clock scheme (chosen over down-counters because it is simpler to make bit-exact).

Ramulator — node.h update_timing / check_ready
// on issuing `cmd` at clock `clk`:
push_history(cmd, clk);
for (edge : timing[level][cmd])
  rc[edge.follow] =
     max(rc[edge.follow],
         history[edge.window-1] + edge.latency);

// legality:
ready = (clk >= rc[cmd]);   // AND over all levels
RTL — timing_engine.sv (per bank shown)
// ready = clk has reached every level's rc
act_ready[b] = (cur_clk >= bnk_act[b]) &
   (cur_clk >= bg_act[g]) & (cur_clk >= pch_act[p]) &
   (cur_clk >= faw_oldest + T_FAW) &
   (cur_clk >= ch_act);

// on ACT issue, reload follows with max():
bnk_rd[b]  <= mx(bnk_rd[b],  T_RCDRD); // 7
bnk_act[b] <= mx(bnk_act[b], T_RC);    // 19
ch_act     <= mx(ch_act, 2);           // channel ACT edge

Two HBM3 subtleties the RTL must reproduce (both later showed up as bugs, §10): the 4-activation window (FAW) — no more than 4 ACTs in any 15 cycles, modeled as a 4-deep shift register of ACT clocks; and the channel 2-cycle ACT edge — after any ACT, the next ACT or PRE on the whole channel waits 2 cycles (HBM3.cpp:305, "ACT happens on its 2nd cycle").

6.4 The scheduler — FR-FCFS (First-Ready, First-Come-First-Served)

Why: across 32 banks and 60+ queued requests, many are blocked by timing each cycle. The arbiter must pick one command that is legal now and makes the most progress. FR-FCFS = prefer any ready command, break ties by oldest arrival.

Ramulator — generic_scheduler.cpp compare()
for each pair (a,b) in buffer:
  if (ready(a) != ready(b))
     keep the ready one;          // First-Ready
  else if (a.arrive <= b.arrive)
     keep a;                      // then FCFS (oldest)
// active_buffer is searched FIRST, then read/write
RTL — hbm3_ctrl_be.sv arbiter
// tier A: active entries, oldest ready (both queues)
for i: if (active[i] & rdy[i] & seq[i]<seqA) bestA=i;
// tier C: non-active, mode-selected queue, oldest ready
for i: if (!active[i] & rdy[i] & seq[i]<seqC) bestC=i;
// pick A if any, else C (with closing guard)
sel = (bestA>=0) ? bestA : bestC;

Note: FCFS uses a monotonically increasing seq stamped at enqueue (equivalent to Ramulator's list/arrival order). Difference: Ramulator searches one combined buffer; the RTL splits into "active" (tier A) and "non-active" (tier C) tiers, which is exactly Ramulator's "active_buffer first, then read/write" order.

6.5 The active buffer — and why it changes who gets enqueued

Why: once a request's ACT has fired, Ramulator moves it out of the read buffer into a separate m_active_buffer. That frees a read-buffer slot, so the frontend can send more requests. This is performance-relevant — and getting it wrong throttles injection (bug #3, §10).

Ramulator — after ACT
if (cmd.is_opening) {          // ACT
  active_buffer.enqueue(req);
  read_buffer.remove(req);    // frees a slot!
}
// accept new send while read_buffer.size() <= 32
RTL — active flag + non-active accept
if (sel_act) rq[i].active <= 1;   // don't move, just flag
// count only NON-active toward the limit:
na_rq = count(valid & !active);
send_accept = (na_rq <= 32) & has_free_slot;

Difference: instead of physically moving entries, the RTL keeps them in place with an active bit and gates send_accept on the non-active count — behaviorally identical, simpler in hardware. (The physical array is enlarged to hold active + non-active entries.)

6.6 Write-mode watermark

Why: switching between reads and writes costs bus turnaround cycles. So the controller stays in read-mode and lets writes pile up, then drains them in a burst — switch to write-mode at 80% full, switch back below 20%.

Ramulator — set_write_mode()
if (!write_mode) {
  if (wq.size() > 0.8*max || rq.size()==0)
     write_mode = true;
} else {
  if (wq.size() < 0.2*max && rq.size()!=0)
     write_mode = false;
}
RTL — write_mode (WHI=25, WLO=6)
if (!write_mode)
  if (na_wq > WHI || na_rq==0) write_mode <= 1;
else
  if (na_wq < WLO && na_rq!=0) write_mode <= 0;

Difference: none — same 0.8/0.2 thresholds against the logical buffer size 32. (Our loads-only traces never enter write-mode, but it is implemented for completeness.)

6.7 Classify-once (hit / miss / conflict statistics)

Why: the hit/miss/conflict counts are the headline performance metric and the thing the bit-exact check compares. Each request must be counted exactly once, from the bank state before its first command takes effect.

Ramulator — update_request_stats (guarded once)
if (!req.is_stat_updated) {
  req.is_stat_updated = true;
  if      (is_row_hit(req))  hits++;
  else if (is_row_open(req)) conflicts++;
  else                     misses++;
}
RTL — classified flag
if (sel_valid & !rq[i].classified) {
  rq[i].classified <= 1;
  case (cls)
    HIT:      stat_hits      <= stat_hits+1;
    CONFLICT: stat_conflicts <= stat_conflicts+1;
    MISS:     stat_misses    <= stat_misses+1;
}

Difference: none — the classified bit is the hardware twin of is_stat_updated.

6.8 Read-data return (read latency)

Why: data does not come back the instant RD issues — it takes nCL+nBL = 11 cycles. The controller tracks in-flight reads and returns them on time.

Ramulator — pending queue
// on RD issue
req.depart = clk + m_read_latency;  // 11
pending.push_back(req);
// each tick
if (pending[0].depart <= clk) callback(pop());
RTL — 11-deep shift register
// on RD issue: drop a token at stage 11
rp_v[READ_LAT] <= 1;  rp_id[READ_LAT] <= id;
// every tick: shift down; stage 0 → rd_resp
for k: rp_v[k] <= rp_v[k+1];
rd_resp_valid <= rp_v[0];

Difference: a FIFO/queue in C++ becomes a fixed-depth shift register in hardware (all reads share the same latency, so order is preserved). Not part of the command stream, so it does not affect the bit-exact command diff.

6.9 Refresh — present in the model, but disabled here

Why it matters: real DRAM must refresh every cell periodically (nREFI=3900 cycles). The twist: Ramulator's AllBank refresh manager loops over "ranks", but HBM3's hierarchy has no rank level, so it issues zero refreshes in this config — verified: the golden command stream contains only ACT/PRE/RD. To match bit-exactly, the RTL therefore does not issue refresh either. (The general-purpose controller variant hbm3_ctrl_top.sv does implement a refresh manager.)

§7Worked examples — cycle by cycle, with real numbers

Using the HBM3_2Gbps timings (tCK = 1 ns): tRCD=7 (ACT→RD), tRP=7 (PRE→ACT), tRAS=17 (ACT→PRE), read latency nCL+nBL=11.

Same byte, three very different costs depending only on what was already in the row buffer. The controller's whole purpose is to turn conflicts and misses into hits — and the RTL reproduces these exact cycle counts because it carries the same tRCD/tRP/tRAS ready-clock arithmetic as Ramulator.

§10Proving it: bit-exact verification & the 3 bugs it caught

A Verilator C++ harness replays Ramulator's own frontend — the 8:3 clock-ratio interleave, send-before-tick ordering, and "stop when all requests are sent" — then diffs the RTL's issued command stream against Ramulator's TraceRecorder golden, command-for-command.

GoldenRamulator + TraceRecorder dumps clk, CMD, pch, bg, ba, row, col per command
Replayharness drives the RTL with the identical frontend injection schedule
Diffcompare every command & cycle, plus hit/miss/conflict totals

The three bugs the diff exposed

Bug 1 — FAW init. The 4-activation-window counters started at 0, so the first ACT was blocked until cycle 15 (it required clk ≥ 0 + tFAW(15)). Ramulator's history window starts "empty". Fix: initialize FAW clocks to −tFAW so the window imposes no block until 4 real ACTs have shifted in. (First ACT moved from clk 15 → clk 1.)
Bug 2 — channel 2-cycle ACT edge. Missing the HBM3.cpp:305 rule that after any ACT the next ACT/PRE on the channel waits 2 cycles. Fix: add a channel-level ch_act ready-clock, reloaded to now+2 on every ACT. (Divergence at command #12 → gone.)
Bug 3 — active-buffer accounting. The frontend was throttled too early because ACTed requests still counted against the read-buffer's 32-entry limit; in Ramulator they move to a separate active buffer. Fix: gate send_accept on the non-active count only. (Divergence at command #71 → gone; this was the last fix to reach full bit-exactness.)
