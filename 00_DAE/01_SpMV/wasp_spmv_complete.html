1. The Real cuSPARSE SpMV Baseline (Blackwell sm_120)

The textbook "for each row → for each nnz" CSR loop does NOT exist in the binary. cuSPARSE uses Merge-Path SpMV (Merrill–Garland, PPoPP'16). Each warp processes exactly 256 nonzeros (8 nnz/lane × 32 lanes) regardless of row boundaries. The per-nonzero body is fully unrolled — no backward branch. This is a fundamentally different kernel structure from the simplified pseudo SASS used in earlier analysis.

2. WASP Applied to SpMV — Pseudo SASS Variants

Important context: The pseudo SASS below uses a simplified textbook CSR model (for-each-row, for-each-nnz inner loop) to illustrate the WASP transformation concept clearly. The real cuSPARSE binary uses merge-path (§1) with fully unrolled, vectorised loads — a more complex baseline. WASP's compiler operates on whatever SASS the original kernel produces; the transformation principles (splitting at LDG load-use boundaries, connecting via RFQ queues) apply to both the textbook model shown here and the real merge-path binary. The WASP paper's evaluation used cuSPARSE as its actual baseline.

2.1 Baseline — cuSPARSE, No Warp Specialization

SINGLE WARP — ALL WORK SERIALIZED (simplified CSR model for illustration)
// Inner loop: for j = row_ptr[i] to row_ptr[i+1]
LOOP_BODY:
    // load col_idx[j] (streaming)
    SHL     Roff, Rj, 2
    IADD    Raddr, Rcol_base, Roff
    LDG     Rcol, [Raddr]              // LDG #1: streaming
    // load values[j] (streaming)
    SHL     Roff, Rj, 3
    IADD    Raddr, Rval_base, Roff
    LDG     Rval, [Raddr]              // LDG #2: streaming
    // gather x[col_idx[j]] — STALLS waiting for Rcol (~400-500 cyc)
    IMAD.WIDE Ra, Rcol, 0x4, Rx_base  // ← RAW dep on LDG #1
    LDG     Rx_val, [Ra]               // LDG #3: indirect gather (~400-500 cyc)
    // multiply-accumulate (NOT fused! separate FMUL + FADD)
    FMUL    Rprod, Rval, Rx_val        // ← STALLS for LDG #2, #3
    FADD    Rsum, Rsum, Rprod
    IADD    Rj, Rj, 32
    BRA     LOOP_TEST

Real binary difference: csrmv_v3 uses descriptor-based LDG.E.EF desc[UR6][R4.64], predicated loads (@!P1 LDG.E.EF), and the body is fully unrolled 8× with no backward branch.

2.2 WASP_COMPILER_TILE — 2-Stage, SMEM Queues (~1.01×)

Only Global→SMEM tile transfers are split. Since cuSPARSE SpMV doesn't use SMEM for its main data path, the gather remains in Stage 1.

Author confirmed: "This is limiting particularly for SpMV as they often don't use SHMEM, and the gather gets serialized."

STAGE 0: stream producer
BAR.WAIT   EMPTY_A            // wait: SMEM buffer free
SHL/IADD   Raddr
LDGSTS     [smem_col], [Raddr]  // col_idx[j]: Global→SMEM
LDGSTS     [smem_val], [Raddr]  // values[j]:  Global→SMEM
BAR.ARRIVE FULL_A             // signal: both col_idx + values ready in SMEM
STAGE 1: gather + compute
BAR.WAIT   FULL_A             // wait: SMEM data ready
LDS        Rcol, [smem_col]   // col_idx[j]: SMEM→RF (extra LDS!)
LDS        Rval, [smem_val]   // values[j]:  SMEM→RF (extra LDS!)
BAR.ARRIVE EMPTY_A            // signal: SMEM buffer consumed
IMAD.WIDE  Ra, Rcol, 0x4, Rx_base // addr for x[col_idx[j]]
LDG        Rx_val, [Ra]       // x[col]: gather (STILL IN THIS WARP)
FMUL       Rprod, Rval, Rx_val // values[j] × x[col]
FADD       Rsum, Rsum, Rprod  // accumulate partial sum

2.3 WASP_COMPILER_ALL — 3-Stage, SMEM Queues (~1.06×)

Author confirmed: "3 stages for SpMV: Index → Gather → Compute/StoreResult."

S0: INDEX PRODUCER
BAR.WAIT  EMPTY_A             // wait: SMEM queue A free
LDGSTS [sq_col],[Ra]          // col_idx[j]: Global→SMEM_A (for S1)
LDGSTS [sq_val],[Ra]          // values[j]:  Global→SMEM_A (for S2, bypasses S1)
BAR.ARR   FULL_A              // signal: both ready in SMEM_A
S1: GATHER PRODUCER
BAR.WAIT  FULL_A              // wait: SMEM_A data ready
LDS   Rcol,[sq_col]           // col_idx[j]: SMEM_A→RF (only col needed by S1)
BAR.ARR   EMPTY_A             // signal: col consumed (values[j] stays for S2)
IMAD.WIDE Ra,Rcol,4,Rx        // compute addr: x_base + col_idx × 4
BAR.WAIT  EMPTY_B             // wait: SMEM queue B free
LDGSTS [sq_xval],[Ra]         // x[col]: Global→SMEM_B (gather result for S2)
BAR.ARR   FULL_B              // signal: x[col] ready in SMEM_B
S2: CONSUMER (compute)
BAR.WAIT  FULL_A              // wait: SMEM_A ready
LDS   Rval,[sq_val]           // values[j]: SMEM_A→RF (from S0, bypassed S1)
BAR.ARR   EMPTY_A             // signal: values consumed from SMEM_A
BAR.WAIT  FULL_B              // wait: SMEM_B ready
LDS   Rx,[sq_xval]            // x[col]: SMEM_B→RF (gather result from S1)
BAR.ARR   EMPTY_B             // signal: x[col] consumed from SMEM_B
FMUL  Rprod, Rval, Rx         // values[j] × x[col]
FADD  Rsum, Rsum, Rprod       // accumulate partial sum
Cross-row overlap (the "bigger deal" per author): Stage 0 can start loading indices for row i+1 while Stage 2 stores y[i]. The memory pipeline never drains between rows. But 2×LDGSTS + 3×LDS + 8×BAR overhead per nnz nearly halves the benefit.

2.4 WASP_GPU + WASP_COMPILER_ALL — 3-Stage, RFQs + HW (~1.08–1.11×)

S0: INDEX (6 regs/thread)
SHL/IADD Raddr
LDG   QS0S1, [Raddr] // col_idx[j]: Global→RFQ (for S1 gather addr)
LDG   QS0S2, [Raddr] // values[j]:  Global→RFQ (for S2, bypasses S1)
S1: GATHER (8 regs/thread)
MOV   Rcol, QS0S1   // col_idx[j]: pop from S0's RFQ
IMAD.WIDE Ra,Rcol,4,Rx // compute addr: x_base + col × 4
LDG   QS1S2, [Ra]   // x[col]: Global→RFQ (gather result for S2)
S2: COMPUTE (24 regs/thread)
MOV   Rval, QS0S2   // values[j]: pop from S0's RFQ (bypassed S1)
MOV   Rx, QS1S2     // x[col]:    pop from S1's RFQ (gather result)
FMUL  Rprod, Rval, Rx // values[j] × x[col]
FADD  Rsum, Rsum, Rprod // accumulate partial sum
No purple (SMEM/barrier) overhead. RFQ scoreboard bits (is_empty/is_full) handle sync in HW. Per-stage register alloc: 6+8+24=38 regs vs 3×24=72 uniform. Pipeline-aware scheduling prioritizes Stage 0.

4. Analytical Model — Why ~1.1× Is the Ceiling

4.1 Per-Nonzero Dependent Chain (single-warp view)

Per-nonzero dependent chain (from real Blackwell SASS):
  LDG col_idx[j]    = ~450 cyc (DRAM round-trip, streaming/coalesced)
  IMAD addr          = ~4 cyc  (FMA pipe)
  LDG x[col]         = ~450 cyc (DRAM round-trip, scattered/indirect)
  FMUL val × x       = ~4 cyc  (FMA pipe)
  FADD partial += p   = ~4 cyc  (FMA pipe)
  ─────────────────────
  Total chain         ≈ 912 cycles per nonzero (single-warp, no interleaving)

Compute fraction: (4+4) / 912 = 0.88%
This is the per-NONZERO latency. Per-ROW and per-SM times are derived below.

4.2 Per-Row Stage Throughput (for pipeline analysis)

For cant (avg 64 nnz/row), each stage's issue-rate throughput — how many cycles to process one row's worth of nonzeros at the instruction issue rate:

Stage throughput per row (cant, 64 nnz/row):

  S0 (index):   64 nnz × 2 LSU issues (LDG col_idx + LDG values) = 128 cycles
  S1 (gather):  64 nnz × 3–6 cyc/nnz (see breakdown below)        = 192–384 cycles  ← bottleneck
  S2 (compute): 64 nnz × 4 issues (2 MOV pops + FMUL + FADD)     = 256 cycles

S1 per-element breakdown — why the range:

  S1 does 3 instructions per nnz with a dependency chain:
    MOV Rcol, QS0S1      ← pop col_idx (1 issue cycle)
    IMAD.WIDE Ra, Rcol   ← compute addr (1 issue, 4-cycle result latency)
    LDG QS1S2, [Ra]      ← issue gather (1 issue, needs Ra ready)

  Strict serialization (6 cyc/nnz):
    Each element waits for IMAD result before issuing LDG.
    Next element's pop cannot start until LDG issues.
    → 6 cyc/nnz × 64 = 384 cycles

  Pipelined within warp (3 cyc/nnz):
    Scheduler overlaps element N+1's pop/IMAD with element N's IMAD wait:
      Cyc 0: MOV col_0  →  Cyc 1: IMAD addr_0  →  Cyc 2: MOV col_1
      Cyc 3: IMAD addr_1  →  Cyc 4: MOV col_2  →  Cyc 5: LDG [addr_0] ✓
      Cyc 6: IMAD addr_2  →  Cyc 7: LDG [addr_1] ✓  →  ...
    → 3 cyc/nnz × 64 = 192 cycles

  Likely reality: ~4 cyc/nnz → ~256 cycles (scheduler imperfections,
  register bank conflicts, and LSU contention reduce ideal pipelining)

  Pipeline throughput = 1 / max(128, 192–384, 256) = 1 row per 256–384 cycles
  Bottleneck: S1 if strict (384), or S1≈S2 if pipelined (192 vs 256)

4.3 Baseline with Warp Interleaving — Why It's Not As Slow As 912 × 64

The baseline (no pipeline) runs on a single warp, but the SM interleaves ~37 warps to hide DRAM stalls. This significantly reduces the effective per-row time.

Baseline per-row time (cant, 64 nnz/row, single warp, NO interleaving):

  Issue streaming loads (col_idx + values):  128 cycles
  + DRAM stall waiting for col_idx:          ~450 cycles (long_scoreboard)
  + Issue IMAD + gather loads:               192–384 cycles (same pipelining
                                             question as S1 — see §4.2)
  + DRAM stall waiting for x[col]:           ~450 cycles (long_scoreboard)
  + Issue FMUL + FADD:                       128 cycles
  ──────────────────────────────────────────
  Useful issue cycles:                       448–640 cycles
  Total (no interleaving):                   ~1,348–1,540 cycles/row

  Using midpoint (~4 cyc/nnz for IMAD+gather): ~544 useful + 900 stall = ~1,444 cycles

With 37-warp interleaving on the SM:

  During a 450-cycle stall, the SM scheduler swaps in other warps.
  But the stalls are NOT fully hidden. Measured: 65% long_scoreboard.

  Why not fully hidden?
  - All 37 warps run the SAME kernel with the SAME stall pattern
  - The dependent chain creates two BACK-TO-BACK stalls (col→gather)
  - The SM has 4 schedulers but 37 warps with bursty issue patterns
  - Occupancy is register-limited at 78% (37.65 of 48 warps)

  Effective baseline per-row time (using 65% stall measurement):
  - 35% of warp-active time is useful (issuing instructions)
  - 65% is stalled on long_scoreboard
  - Effective time = useful_cycles / useful_fraction
  - Using midpoint: 544 / 0.35 ≈ ~1,550 warp-cycles per row
  - Using strict:   640 / 0.35 ≈ ~1,830
  - Using pipelined: 448 / 0.35 ≈ ~1,280

  All estimates are high because 65% stall means interleaving barely helps.
  37 warps with identical stall-heavy patterns can't fill each other's gaps.

4.4 Theoretical vs. Actual Speedup

Naïve pipeline speedup (ignoring warp interleaving changes):

  Using midpoint estimates:
    Baseline effective: ~1,550 warp-cycles/row
    Pipeline bottleneck: ~256–384 cycles/row (S1, depending on pipelining)

  Speedup = 1,550 / 320 ≈ 4.8× (WRONG — overly optimistic)

Why this overpredicts:

  1. The pipeline ALSO has stalls. S1 issues scattered gather loads that
     hit DRAM at ~450 cyc. S1's 256–384 cyc is the ISSUE throughput — the
     actual wall-clock is longer because S1 stalls on its own gathers.

  2. The baseline already hides SOME latency via 37-warp interleaving.
     The pipeline changes the warp mix (3 specialized warps instead of 1)
     which changes the interleaving dynamics — not necessarily better.

  3. Pipeline overhead: control flow replication (~18% extra instructions),
     register pressure (1.26× footprint with per-stage alloc), and
     RFQ space competing with registers.

More realistic model (accounting for S1's own stalls):

  S1 also stalls on DRAM (its gather loads take ~450 cyc).
  With pipeline interleaving between S0/S1/S2 on the same SM,
  some of S1's stalls are hidden by S0/S2 work — but not all.

  Estimated S1 effective time ≈ issue_time × (1 / useful_fraction)
  If S1's stall pattern is similar to baseline (~50–60% stall):
    S1 effective ≈ 320 / 0.45 ≈ 710 cycles/row (midpoint)
    Range: 192/0.45 ≈ 427  to  384/0.45 ≈ 853

  Pipeline speedup ≈ 1,550 / 710 ≈ 2.2× (upper bound)

  With ~20% overhead: 1,550 / (710 × 1.2) ≈ 1.8× (optimistic ceiling)

  Further reduced by cant's 89% BW saturation (can't push more bytes
  through DRAM even with better MLP):
  Realistic ceiling ≈ 1.2–1.4×

  Actual WASP: 1.08–1.11× (additional losses from imperfect
  overlap, queue dynamics, changed occupancy, not all matrices
  benefit equally)

Where WASP's actual 1.08–1.11× comes from:

  NOT from compute-memory overlap (contributes <1%)
  Mainly from:
  - Cross-row overlap: S0 starts next row while S2 finishes current
    → eliminates the "drain" gap between rows
  - Better MLP: pipeline keeps more memory requests in-flight
  - Better BW utilization: spreading requests over time + DRAM row reuse
  - Minus: overhead, register pressure, changed occupancy
From Blackwell ncu measurements: cant matrix reaches 89.27% of peak DRAM bandwidth and 65.28% long_scoreboard stalls. This means SpMV is simultaneously bandwidth-saturated AND latency-bound. WASP's MLP extraction (cross-row overlap) helps the latency-bound aspect, but on matrices that are already BW-saturated, the ceiling is even lower.

The author confirmed: "First question is whether we are saturating DRAM bandwidth already. If not, the assumption is that there isn't enough MLP, so we can use warp specialization and WASP."

5. Beyond WASP — Where Real Speedup Comes From

After the author's response and real hardware metrics, the assessment is: the overlap opportunity is largely exhausted by WASP. Remaining gains must attack different bottlenecks.

#TechniqueSW/HWWhat it attacksEstimated gain
①Prefetch warp (cache warming)HW+SWT_gather directly — warm cache for x[col] before load warp needs it~1.1–1.3×
②On-the-fly compressionHW+SWBytes transferred — delta-encode col_idx, quantize values~1.1–1.2×
③Column-sorted blockingSW onlyCache hit rate for x vector — cluster similar col indices~1.05–1.15×
④HW coalescing bufferHWBW efficiency — merge irregular x[col] requests across warps~1.05–1.1×
⑤Work-stealing schedulerSWLoad imbalance — dynamic row redistribution~1.02–1.05×
Key insight: The prefetch warp is fundamentally different from WASP's DAE model. It doesn't produce data for consumers — it produces cache state. The gather stage's latency drops from ~450 cyc (DRAM) to ~30 cyc (L1 hit), directly shrinking the bottleneck stage. This is the only path to >1.2× on SpMV.
Revised combined estimate: ~1.3–1.6× over baseline (down from earlier 1.8–2.2× estimate, after learning from the author that WASP already captures most MLP/overlap benefit, and from Blackwell measurements showing cant is already 89% BW-saturated).
