1. The Sparse Matrix and CSR Storage

A sparse matrix stores only nonzero elements. The CSR format uses three arrays:

The key problem is visible in the matrix: row 3 has 5 nonzeros while row 4 has zero. Naive row-per-thread assignment leads to massive load imbalance. cuSPARSE solves this with the merge-path algorithm.

2. Merge-Path: Equal-Sized Warp Tiles

Merge-path (Merrill–Garland, PPoPP'16) transforms the 2D (rows × nnz) iteration space into a 1D "merged coordinate" array. Each position is either a row step (↓) or a nonzero step (→). This 1D array is sliced into equal tiles of 256 nonzeros, each assigned to one warp:

RED.E.ADD cant: 4M nnz ÷ 256 = 15,656 warp tiles = 3,914 blocks × 4 warps/block

A companion csr_partition_kernel (72 instructions, trivially fast) binary-searches row_ptr to precompute the starting row for each warp tile. The main csrmv_v3_kernel reads this partition table and processes its 256 nonzeros.

3. The Dependent Load Chain — The Fundamental Bottleneck

The inner loop has a fundamental problem: x[col] cannot be loaded until col_idx[j] returns from DRAM, serializing two long-latency memory accesses. The kernel batches 8 iterations per lane to overlap the first halves, but the col_idx → x[col] dependency is inescapable.

The fully-unrolled Phase 4 in real SASS — 4 interleaved groups

In the csrmv_v3 binary, Phase 4 (PCs 0x0170–0x1300, ~430 instructions) is fully unrolled — no backward branch wraps the per-nonzero body. The compiler does NOT generate 8 sequential iterations of the same chain. Instead, it deeply interleaves all 8 batches across 4 functional groups to maximise memory-level parallelism:

GROUP 1: FIRE ALL STREAMING LOADS (PCs 0x0170–0x02b0)

All 8 batches of col_idx[] loads + all 8 batches of values[] loads fire back-to-back. No data dependency between batches — maximum MLP. All ~16 loads are in-flight to DRAM simultaneously.

── col_idx loads (one warp-width per batch, offset +0x80 = 32 lanes × 4B) ──
0x0170: @!P1 LDG.E.EF R35, desc[UR6][R4.64]         ← batch 0: col_idx[nnz 0..31]
0x01a0: @!P4 LDG.E.EF R23, desc[UR6][R4.64+0x180]   ← batch 3: col_idx[nnz 96..127]
0x01b0: @!P5 LDG.E.EF R21, desc[UR6][R4.64+0x200]   ← batch 4: col_idx[nnz 128..159]
0x01d0: @!P2 LDG.E.EF R29, desc[UR6][R4.64+0x80]    ← batch 1: col_idx[nnz 32..63]
0x01e0: @!P3 LDG.E.EF R25, desc[UR6][R4.64+0x100]   ← batch 2: col_idx[nnz 64..95]
        (batches 5–7 at higher offsets follow in subsequent PCs)

── values loads (same structure, different base register R4 with offsets) ──
0x0280: @!P1 LDG.E.EF R20, desc[UR6][R4.64]         ← batch 0: values[nnz 0..31]
0x0290: @!P2 LDG.E.EF R32, desc[UR6][R4.64+0x80]    ← batch 1: values[nnz 32..63]
0x02a0: @!P3 LDG.E.EF R26, desc[UR6][R4.64+0x100]   ← batch 2: values[nnz 64..95]
0x02b0: @!P5 LDG.E.EF R24, desc[UR6][R4.64+0x200]   ← batch 4: values[nnz 128..159]
0x0300: @!P4 LDG.E.EF R8,  desc[UR6][R4.64+0x180]   ← batch 3: values[nnz 96..127]
        (batches 5–7 follow)

Loads are predicated (@!P1..@!P5) for bounds checking — tiles near the end of the matrix may have fewer than 256 valid nonzeros. All loads fire within ~20 cycles. Batch ordering is NOT sequential (0,3,4,1,2...) — the compiler reorders to minimize register-file bank conflicts.

GROUP 2: ADDRESS COMPUTATION + DEPENDENT GATHERS (PCs 0x02c0–0x0370)

As col_idx results return from DRAM (~400–500 cycles later), compute gather addresses and issue dependent x[col] loads. ⚠ THIS IS WHERE THE WARP STALLS — long_scoreboard (65%).

── IMAD.WIDE: x_base + col_idx × 4 (needs col_idx result → STALLS!) ──
0x02c0: @!P4 IMAD.WIDE R16, R23, 0x4, R16           ← batch 3: addr(x[col])
0x02d0: @!P5 IMAD.WIDE R22, R21, 0x4, R8            ← batch 4: addr(x[col])
0x02f0: @!P1 IMAD.WIDE R30, R35, 0x4, R30           ← batch 0: addr(x[col])
0x0310: @!P2 IMAD.WIDE R14, R29, 0x4, R14           ← batch 1: addr(x[col])
0x0330: @!P3 IMAD.WIDE R18, R25, 0x4, R18           ← batch 2: addr(x[col])

── LD.E: x[col] — the dependent gather loads (note: LD.E not LDG.E.EF) ──
0x0320: @!P1 LD.E R11, desc[UR6][R30.64]            ← batch 0: x[col_0] (scattered)
0x0340: @!P2 LD.E R15, desc[UR6][R14.64]            ← batch 1: x[col_1]
0x0350: @!P3 LD.E R18, desc[UR6][R18.64]            ← batch 2: x[col_2]
0x0360: @!P4 LD.E R16, desc[UR6][R16.64]            ← batch 3: x[col_3]
0x0370: @!P5 LD.E R22, desc[UR6][R22.64]            ← batch 4: x[col_4]
        (batches 5–7 follow)

The compiler interleaves IMAD.WIDE and LD.E across batches: while batch 0's col_idx is still in-flight, batch 3's may have already returned. Each LD.E is a scattered, indirect load — these cause the 65% long_scoreboard stalls.

GROUP 3: MULTIPLY (PCs 0x03e0–0x04a0)

Once values[j] and x[col] are in registers, compute val × x[col] using two separate FMULs (NOT FFMA). The first FMUL multiplies by the alpha coefficient; the second multiplies by x[col].

── FMUL pass 1: values[j] × alpha (R7 = alpha, reused across all batches) ──
0x03e0: @!P1 FMUL R20, R20, R7.reuse                ← batch 0: val × alpha
0x03f0: @!P2 FMUL R32, R32, R7.reuse                ← batch 1: val × alpha
0x0400: @!P3 FMUL R13, R26, R7.reuse                ← batch 2: val × alpha
0x0410: @!P5 FMUL R19, R24, R7.reuse                ← batch 4: val × alpha
0x0430: @!P4 FMUL R17, R8,  R7                      ← batch 3: val × alpha

── FMUL pass 2: (val×alpha) × x[col] (the actual SpMV product) ──
0x0450: @!P1 FMUL R12, R11, R20                     ← batch 0: (val×α) × x[col_0]
0x0470: @!P2 FMUL R11, R15, R32                     ← batch 1: (val×α) × x[col_1]
0x0480: @!P3 FMUL R4,  R18, R13                     ← batch 2: (val×α) × x[col_2]
0x0490: @!P4 FMUL R5,  R16, R17                     ← batch 3: (val×α) × x[col_3]
0x04a0: @!P5 FMUL R14, R22, R19                     ← batch 4: (val×α) × x[col_4]
        (batches 5–7 follow)

Two separate FMULs, not FFMA — the compiler chose split multiply because FMUL+FADD allows more scheduling freedom. The FADD is deferred to Group 4 where it's interleaved with row boundary detection.

GROUP 4: ROW BOUNDARY DETECTION + ACCUMULATION (PCs 0x04b0–0x1300, ~350 instructions)

The largest and most complex group. Per-lane binary search to find which row owns each nonzero, using SHFL.IDX across warp lanes. Short backward-BRA loops walk row boundaries in shared memory. FADD accumulates products per row segment. This is NOT a simple "FADD per nnz" — the structure depends on how many rows the tile spans.

── Row boundary detection via SHFL.IDX (binary search, 5 steps = log2(32)) ──
0x05e0: SHFL.IDX PT, R8,  R16, 0x10, 0x1f          ← bisection stride=16
0x0620: SHFL.IDX PT, R20, R16, R15, 0x1f           ← bisection stride=8
0x0680: SHFL.IDX PT, R20, R16, R17, 0x1f           ← bisection stride=4
0x06d0: SHFL.IDX PT, R18, R16, R19, 0x1f           ← bisection stride=2
0x0730: SHFL.IDX PT, R20, R16, R21, 0x1f           ← bisection stride=1

── STS/LDS: stage row boundary info + partial sums in shared memory ──
0x0750: STS [R19], R16                              ← write row boundary to smem
0x07a0: LDS R18, [R18+0x4]                          ← read next boundary from smem
0x0840: LDS R20, [R16+0x8]                          ← read during boundary walk
0x0930: LDS R25, [R16+0x8]                          ← repeated for subsequent rows

── Row boundary walk loops (the ONLY backward BRAs in Phase 4) ──
0x0860: @!P0 BRA 0x820                              ← walk row boundary 1 (short loop)
0x0950: @!P0 BRA 0x910                              ← walk row boundary 2
        ...repeats for each row crossing within this tile (~4 for cant)...

── FADD accumulation (per row segment, deferred from Group 3) ──
        FADDs accumulate products for each row segment.
        The reduction structure depends on how many rows this warp tile spans.
        Pattern repeats for each of the ~4 row boundaries through PC 0x1300.

Group 4 is by far the largest (~350 of ~430 Phase 4 instructions) because the row boundary detection and reduction logic has many branches depending on how many rows the tile spans. For cant (avg 64 nnz/row), each warp tile spans ~4 rows.

Key structural insight: Phase 4 is NOT "8 identical iterations." The compiler organizes it as: (1) one big batch of streaming loads for maximum MLP → (2) one big batch of dependent gathers, interleaved as col_idx results arrive → (3) one big batch of FMULs → (4) one complex row-boundary section with SHFL binary search, STS/LDS shared memory staging, short backward-BRA loops, and per-row-segment FADD accumulation. The program order interleaves LSU and FMA pipes: LSU→LSU→FMA→LSU→FMA→FMA→LSU→FMA — not "all LSU then all FMA."

4. The 6 Phases of csrmv_v3_kernel

Each warp executes 6 sequential phases. Phase 4 (per-nonzero processing) dominates execution time.

PHASE 0: SETUP (PC 0x0000–0x0220, ~20 cycles)

Load kernel parameters from constant memory (LDC, LDCU), compute warp_id, lane_id. S2R reads special registers. Constant cache hits at ~4 cycles. Not on critical path.

warp_id = blockIdx.x * 4 + (threadIdx.x >> 5)    // 4 warps per block
lane_id = threadIdx.x & 31
warp_nnz_start = warp_id * 256                    // 256 nnz per warp tile
PHASE 1: LOAD PARTITION TUPLE (PC 0x0220–0x02e0)

Read pre-computed partition data from csr_partition_kernel: the starting and ending row for this warp tile.

(row_lo, nnz_lo) = partitions[warp_id]
(row_hi, nnz_hi) = partitions[warp_id + 1]
PHASE 2: ROW_PTR WALK (PC 0x1310–0x1440, backward BRA)

Walk row_ptr from row_lo to row_hi, staging segments into shared memory. One of only two backward branches in the kernel.

PHASE 3: SHARED-MEM BINARY SEARCH (PC 0x0820–0x0bc0)

Each lane bisects smem_row_ptr to find which row owns its nnz_index. Uses SHFL.IDX and LDS. 4 backward BRAs for bisection steps.

my_nnz_idx = warp_nnz_start + lane_id * 8    // 8 nnz per lane
my_row     = bsearch(smem_row_ptr, my_nnz_idx)
PHASE 4: FULLY-UNROLLED PER-NONZERO PROCESSING (PC 0x0170–0x1300, ~430 instructions)

This is where the warp spends most of its time. The compiler interleaves all 8 batches across 4 functional groups (see §3 above). The per-nonzero instruction breakdown in program order (not sorted by pipe):

StepPer nonzeroPipeProgram order notes

Total per nnz5 LSU + 3 FMA = 8 instructions — pipe order: LSU→LSU→FMA→LSU→FMA→FMA→LSU→FMA
PHASE 5: WARP-LEVEL REDUCTION (PC 0x1dd0–0x1e80)

SHFL.DOWN butterfly across 32 lanes for each row segment. 5 steps: stride 16→8→4→2→1. Lane 0 gets the final sum for each row owned by this warp.

PHASE 6: OUTPUT WRITE (PC 0x1700–0x36cc)

Interior rows: STG.E to y[row]. Boundary rows (spanning warp tiles): RED.E.ADD or ATOM.E.ADD.F32.FTZ.RN atomic-add. Includes CAS-spin paths for boundary publication.

6. Warp Execution Timeline — Stall Analysis

A single warp's life through the 6 phases. The rhythm is: issue burst (fire independent instructions at 1/cycle) → stall (hit register dependency on an outstanding DRAM load, ~450 cycles away) → swap (scheduler picks another of the ~37 resident warps) → resume (DRAM returns).

a LDG col_idx[j] — batch of 8 vectorised global loads for column indices. Issues back-to-back at 1/cycle. (LSU pipe) b STALL ~400–500 cyc (long_scoreboard) — warp blocked waiting for col_idx to return from DRAM. Scheduler swaps in another warp. c IMAD.WIDE addr — compute gather address: x_base + col × 4. Fires immediately once col_idx register is ready. (~4 cyc, FMA pipe) d LDG x[col] — dependent gather load. Address was computed in (c). Issues to DRAM. (LSU pipe) e STALL ~400–500 cyc (long_scoreboard) — warp blocked waiting for x[col] to return from DRAM. Scheduler swaps again. f FMUL val × x[col] — per-nonzero multiply. Fires once both val and x[col] are in registers. (~4 cyc, FMA pipe) g FADD partial += product — per-nonzero accumulate. Separate from FMUL (not fused). (~4 cyc, FMA pipe)
LDG STALL AGEN FMUL/FADD

The pattern a→b→c→d→e→f→g repeats for each nonzero. 65% of warp-cycles are spent in (b) and (e) — the long_scoreboard stalls. The SM interleaves ~37 warps to hide this latency, but cannot fully fill the pipeline because occupancy is register-limited at 78%.

Why LSU 29% and DRAM 89% don't contradict: They measure different things with different denominators. LSU% = issue-slot utilisation (how often an LSU instruction is dispatched, normalised against per-pipe peak × active cycles). DRAM% = byte throughput utilisation (how full the memory bus is, normalised against peak bandwidth × elapsed cycles). A single vectorised LDG.E.128 issues once but pulls 16B per lane × 32 lanes × queued-in-flight requests. Few issues, many bytes — that's exactly what an LSU 29% / DRAM 89% gap means. The two denominators also differ in time basis (active vs elapsed).
The double bind: SpMV on dense-row matrices like cant is simultaneously DRAM-bandwidth-bound (89% of peak) and memory-latency-bound (65% long_scoreboard). Adding more warps doesn't help (occupancy is at the register-limit ceiling). Shrinking the per-nnz dependent chain is the only lever: the gather load on x[col_idx] determines everything.

8. The x-Vector: The Shared Bottleneck

The CSR arrays (values, col_idx, row_ptr) are partitioned cleanly — each warp reads its own non-overlapping slice, coalesced and streaming. But every warp on every SM reads from the same x-vector, and the access pattern is determined by column indices — essentially random.

9. Where the Time Goes — Cycle Budget Summary

Bottom line for researchers: SpMV's arithmetic intensity is ~0.17 FLOP/byte. The FMA pipe runs at 3.34% utilisation. The kernel is 99% memory-bound. Any acceleration technique that works by "overlapping compute with memory" (including warp specialization / WASP) has a maximum theoretical benefit of <1% from the overlap itself. The only path to meaningful speedup is reducing the memory bottleneck directly: fewer bytes, lower latency, better cache hits.
