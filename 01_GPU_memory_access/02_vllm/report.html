vLLM Serving: Weight & KV Cache Memory Access

The vLLM counterpart to the PyTorch/HF studies in this section. Same models (Qwen2.5-1.5B, GPT-OSS-20B), same RTX 5080, traced live under NVBit — but served through vLLM 0.23 instead of eager PyTorch, so the paged KV cache and fused/MoE kernels show up in the access pattern. Every number is from a real vLLM forward with addresses attributed to the actual weight / KV tensor by data_ptr() range.

vLLM 0.23 · torch 2.11 · cu130 RTX 5080 · Blackwell sm_120 LIVE NVBit trace · enforce_eager · CUDA_MODULE_LOADING=LAZY

§1Summary

The cache conclusion from the PyTorch studies survives the move to a production serving stack — KV streams, weights are tiled — but vLLM's design choices (paged KV, MoE expert kernels) leave a measurable fingerprint.

Three findings. (1) KV cache still streams under vLLM (~0.95 at 1KB), but its paged cache breaks long runs at block boundaries — visible as a 16KB-contiguity drop the contiguous PyTorch cache doesn't have. (2) vLLM weights are more 1KB-contiguous than HF (bigger contiguous tiles per GEMM). (3) The GPT-OSS weight "gap" vs PyTorch is not a quantization difference — both are MXFP4; it was dense-projection (HF) vs MoE-expert (vLLM) weights, and the matched expert trace exposed a real consumption-path difference (§4).

§3KV cache: paging is visible in the data

Both stacks keep the KV cache streaming at 1 KB (~0.95 vLLM, ~0.99 PyTorch). The difference is at coarser granularity:

  • vLLM is consistently lower, and the gap widens at 16 KB (Qwen 0.667 vs 0.921). vLLM's KV is paged: K/V is contiguous within a fixed token block (written by reshape_and_cache_kernel_flash) but runs break at block boundaries.
  • PyTorch keeps a contiguous past_key_values (CatArrayBatchedCopy), so it stays contiguous over much larger spans — hence the higher 16 KB number.
  • Tile coverage stays high on both (~94–100%): when a KV region is touched, nearly the whole tile is read, so block-granular KV compression remains viable on both stacks.

So paging doesn't break KV streaming — it pins the natural KV block to vLLM's page size.

Page size is visible: contiguity breaks where the page ends

Sweeping the contiguity metric past 16 KB (granularities 32 KB and 64 KB added to the analyzer) shows the break tracking each model's KV page size — 16 KB for Qwen, 32 KB for GPT-OSS. The page is 16 tokens wide, so its byte size is 16 × kv_heads × head_dim × 2 B × 2(K,V) per layer (Qwen: 2 heads × 128 = 16 KB; GPT-OSS: 8 heads × 64 = 32 KB).

Read the 16 KB column: Qwen is exactly at its page boundary and contiguity has dropped to 0.667, while GPT-OSS — where 16 KB is only half a 32 KB page — still holds at 0.855. That cross-model contrast is the page size showing through the access pattern.

Read 32–64 KB with caution. The touched KV footprint is small, so the number of blocks collapses at coarse granularity (Qwen 64 KB → 26 blocks; GPT-OSS 64 KB → 24). With only a couple dozen points the metric gets noisy — GPT-OSS is even non-monotonic (8 KB below 16 KB) and Qwen's 64 KB = 0.000 is a small-sample edge, not a true zero. The robust signal is directional: Qwen loses contiguity by 16 KB, GPT-OSS holds through 16 KB and only erodes past 32 KB — exactly their page sizes. (This is why 16 KB was the original sweep ceiling.)

§4Matched MXFP4 experts — isolating framework from quantization

The raw GPT-OSS weight gap (PyTorch 0.999 vs vLLM 0.360 at 16 KB) looks dramatic, so we chased it down. It is not a quantization effect:

  • Both stacks already run GPT-OSS in MXFP4 (4-bit experts, ~13.8 GB). The HF run loads 96 MXFP4 expert tensors; vLLM serves the same 4-bit checkpoint via marlin_moe_wna.
  • The gap was apples-to-oranges: the PyTorch row measured dense projection weights (gemvx/cutlass), the vLLM row measured MoE expert weights (marlin_moe). Different weights, not different precision.

Running the matched expert-vs-expert trace (both MXFP4 MoE experts) then exposed a genuine framework difference — in how the experts are consumed:

Stack · expert kernelGlobal-memory weight stream?weight-attributedp_next_1 (1KB / 16KB)

The real finding. vLLM's Marlin kernel streams the packed 4-bit expert weights straight from global memory (attributable, tiled). HF's triton grouped-GEMM stages experts through shared memory + transient repacked buffers — even at 600k traced records its global traffic is 17.3M shared/local accesses with no captured packed-weight stream. So a like-for-like expert p_next_1 exists for vLLM but not HF: the MXFP4 consumption path differs by framework, the data format does not.

§6Setup & how it was traced

vLLM 0.23 had to be installed in an isolated venv against torch 2.11 + cu130 to run on Blackwell (sm_120) at all — the stock build was ABI-broken. For a faithful trace it must run in-process and eager, with lazy module loading on:

# env recipe baked into the vLLM workload
VLLM_ENABLE_V1_MULTIPROCESSING=0   # model in the traced process (range capture + NVBit see it)
VLLM_USE_FLASHINFER_SAMPLER=0      # native sampler (flashinfer JIT mis-detects arch)
CUDA_MODULE_LOADING=LAZY           # EAGER stalls vLLM loading every cubin (flashinfer/cutlass)
TORCH_CUDA_ARCH_LIST=12.0
enforce_eager=True                 # no CUDA graphs to hide from NVBit
  • Range capture: exact model.named_parameters() (weights) and model_runner.kv_caches (the paged pool) from the in-process engine — high confidence.
  • GPT-OSS-20B on 16 GB is tight: needs gpu_memory_utilization=0.93 and max_num_seqs=1 to leave room for the KV pool and NVBit under MXFP4.
  • Per-category targeted traces (weight-only / KV-only regex) avoid record-budget splitting across kernels.

§7Caveats

  • HF expert weights are not directly observable. The triton MXFP4 grouped-GEMM stages through shared memory, so its packed-weight global stream couldn't be attributed — that absence is itself §4's finding, but it means there is no HF expert p_next_1 to compare numerically.
  • PyTorch rows are the existing dense/kvcopy traces from the non-vLLM studies in this section; vLLM rows are fresh single-category traces. Methodology (targeted regex, NVBit attribution) is identical.
  • Prompt/decode lengths differ slightly per run; contiguity metrics are pattern properties, largely insensitive to sequence length.
