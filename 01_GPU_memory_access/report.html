1 · The methodology

The earlier metric asked a yes/no-ish probability at a fixed block size (“is the access contiguous at 1 KB?”). This one is more direct: take every address a kernel touched for a category, sort the unique addresses, and cut them into maximal contiguous runs — a run ends wherever the gap to the next touched address exceeds a tolerance T. Each run’s size is last − first; we histogram the run sizes.

Real access is never byte-perfectly contiguous (a vectorized load already skips ~16 B), so a gap tolerance is unavoidable. Sweeping T from 0 to 4 MB revealed the run-size distribution sits on flat plateaus separated by the natural gaps in the data (element stride → GEMM tile pitch → tensor boundary). That gives two physically distinct, robust operating points:

T = 256 B
Access — above element/sub-head strides, below the ~1.5 KB tile pitch. Answers “how is it read?”
T = 256 KB
Footprint — above the GEMM tile-block strides (which scale with matrix size), below MB tensor boundaries. Answers “how big is the contiguous tensor?”

Both values sit inside a stability plateau (run count is flat across a wide range of T), so the result doesn’t depend on the exact constant. Each cell is self-verified: attribution > 0, plateau stability, and whether the footprint run-count reproduces the ground-truth tensor count from the workload’s own data_ptr() ranges.

2 · Limitations

  • Sampling. NVBit captures a per-kernel record budget, so a run can be truncated if its tensor wasn’t fully traced. Weights are sampling-sensitive — these runs use 200 k records/grid, 2 M total; KV is fine at default budget. The footprint view needs enough density that intra-tensor holes stay below 256 KB; for very large matrices that bound can fail (we verify it per cell).
  • The gap tolerance is a modeling choice. The two plateaus are robust, but the 1–8 KB transition between them is deliberately avoided; numbers inside it are unstable.
  • Ground-truth capture is partial for KV. The workload records device ranges via tensor introspection; framework-internal per-layer KV tensors are only partially named, so the KV “ground-truth tensor count” undercounts and won’t equal the run count even when the measurement is correct (see the ✓/≠ tags).
  • Two quantized-weight cells are imperfect (documented, never silently dropped): vLLM+int4 (bitsandbytes) relocates its packed weights after our range capture → a true gap; PyTorch+MXFP4 experts stage through shared memory → only the dense projections are visible.

The precision twist. Mistral is int4 (bitsandbytes NF4). On PyTorch the weight histogram is not thousands of 1.5 KB tiles — it is 10 runs of ~1 MB. KV is unquantized and streams normally on both stacks.

4 · Cross-cutting analysis — how, why, and where it matches ground truth

Weights are tiled — and the tile size is set by the consumption kernel, not the model. A bf16 GEMM reads a weight matrix in tiles, contiguous for ~1.2–1.7 KB (one tile-row) before jumping a row-pitch — hence tens of thousands of ~1.5 KB access runs. Quantization changes this completely: int4 bitsandbytes reads the packed weights as ~1 MB linear streams (the dequant kernel sweeps them to expand to bf16), and MXFP4 marlin reads packed experts in ~69 KB blocks. Three precisions → three granularities, all measured directly.

Why the footprint matches ground truth. Raise the tolerance to 256 KB and the tile-pitch and tile-block gaps are bridged, so each whole matrix becomes one run. The footprint run count then equals the number of weight tensors the workload allocated (6, 8, 10, … — the ✓ tags), because the GEMM really does cover each matrix contiguously; only the order of traversal (tiled) hid it at the access scale. This is the strongest validation: the same addresses, viewed at two tolerances, recover both the access pattern and the true allocation.

KV streams; paging is visible as smaller runs. PyTorch’s contiguous past_key_values shows up as a few huge runs (up to 11.5 MB — whole multi-layer spans), because the cache-append kernel writes it linearly. vLLM’s paged KV shows up as dozens of mid-sized runs (84 KB–1.9 MB): a single decoded sequence gets contiguous blocks, so it is far less fragmented than the textbook “16 KB page” intuition, yet still measurably chunkier than the monolithic PyTorch cache.

Where it does not match — and why that’s expected. For KV, the footprint run count rarely equals the ground-truth tensor count: PyTorch merges adjacent per-layer tensors into bigger runs (run count < tensor count), while vLLM has a single pooled tensor (ground-truth = 1) split into many touched block-runs (run count ≫ 1). Both are correct physics, not measurement error — they are flagged ≠ rather than hidden. The two quantized-weight blemishes (vLLM+int4 gap, PyTorch+MXFP4 experts) are consequences of how those stacks physically place/consume weights, confirmed by address analysis.

One-line takeaway. Weights are tiled into kernel-sized chunks (bf16 ~1.5 KB, MXFP4 ~69 KB, int4 ~1 MB streams) that live inside a handful of large contiguous tensors; KV is streamed — monolithic under PyTorch, paged-but-contiguous under vLLM. The access/footprint split holds on every architecture and precision tested.
