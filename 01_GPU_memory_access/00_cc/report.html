Do LLM Weights & KV Cache Stream?

Live-trace results only. Every number here comes from a NVBit trace of a real PyTorch forward, with each memory address attributed to the actual weight / KV-cache tensor by its data_ptr() range — the literal production process, no proxy kernels.

LIVE PyTorch trace · CUDA_MODULE_LOADING=EAGER RTX 5080 · Blackwell sm_120 · CUDA 13.0 GPT-2 (fp32)

§1Summary

The question: when a cache line of a weight tensor or the KV cache is touched, how likely is a neighboring line to be touched too? — i.e. do these tensors stream (large contiguous, fully-used blocks) or random-access? Decides whether large-block, high-ratio, pipelined HBM decompression is worthwhile.

✓ On the real process: weights and KV cache both STREAM

So what? Classic cache-line compressors (BDI, FPC, C-Pack) assume random, latency-sensitive access. The live trace shows the opposite — fully contiguous, fully-block-covered, streamed once per token, latency-amortizable. Ideal for large-block, high-ratio, pipelined decompression in an HBM logic die. Recommended block: 8–16 KB weights, 4 KB KV.

Random-vs-Streaming verdict matrix (real tensors)

Hover any cell for the underlying numbers. Streaming = P(i+1) ≥ 0.5 and median contiguous run ≥ 4 lines.

Paged KV is the one nuance. vLLM's real paged KV (last row) streams cleanly within each 4 KB physical block (Streaming at 32 B–1 KB, coverage 1.0 to 4 KB) but flips to Random at 8 KB — the scattered block table makes the sweep jump at every block boundary. This doesn't weaken the case; it pins the natural KV compression block to the paged block size (4 KB), where coverage is still 1.0.

Scale & MoE hold up. The verdict survives a 20B Mixture-of-Experts model with 4-bit (MXFP4) experts — gpt-oss-20b, traced live on the 16 GB RTX 5080. Its MoE expert GEMMs read the packed 4-bit weight matrices contiguously (P(i+1)=1.0, coverage ≈1.0 to 16 KB), exactly like the dense bf16/fp32 models. Streaming is not an artifact of small dense models.

§2How it was traced

The guide's preferred recipe — trace a real forward and map every address to a weight/KV data_ptr() range — works here once lazy CUDA module loading is turned off. (An earlier study wrongly called PyTorch un-traceable; the cause was CUDA_MODULE_LOADING=LAZY hiding kernels from NVBit, not the driver.) With EAGER, NVBit captures the full forward and, because we record the tensor ranges in the same process, attribution is exact.

Attribute by rangeEach address → weight / KV / other. The real fmha kernel reads KV; cutlass/gemvx read weights.
Stream metricsPer role: line/sector/tile IDs → run length, P(i+N), tile coverage, reuse distance at 32 B…8 KB.
  • Granularity IDs: line_id = addr >> k, k = 5/6/7/10/12/13 for 32B/64B/128B/1KB/4KB/8KB.
  • Run length = longest contiguous span of touched line IDs (are blocks fully populated?).
  • Tile coverage = fraction of 32 B sectors touched inside each tile (are fetched blocks wasted?).
Footprint vs. access order — a clarification. The "Streaming" verdict here is about the footprint: is a tensor a contiguous, fully-used block of memory? That is what decides block-compressibility, and it is what the metrics above measure. It is not a claim that a kernel reads the bytes in strict linear order. A weight can be contiguous in storage yet read in tiled / column order by the matmul.

We verified this on a real gpt-oss-20b dense weight (the q_proj Linear, [4096, 2880] bf16). It is stored fully contiguous (stride = (2880, 1), is_contiguous = True) and its 23.6 MB footprint is swept 100% — so it compresses as a block. But the CUTLASS kernel (cutlass_80_wmma_tensorop_bf16…tn, which reads W transposed for x·Wᵀ) walks it in tiles with column-like strides: contiguous/vectorized within a tile (4 B / 16 B loads), then jumps of exactly 1×, 2×, and 16× the 5760 B row pitch to step across output rows. So the access order is tiled even though the layout is contiguous — that affects prefetch / streaming-decompression design, not whether large-block compression is viable.

§6Caveats

  • Trace volume & short context. The literal trace costs ≈7 GB per token for GPT-2 (49.8 GB for Qwen), so each run is scoped to a short context (GPT-2 seq = 64, Qwen seq = 8). At seq = 8 the Qwen KV cache is tiny (~2 KB per head), so its coarse-tile (4–16 KB) coverage and single 4 KB verdict are small-footprint artifacts; the streaming signal is clear at 32 B–1 KB.
  • Pre-coalescer virtual addresses. NVBit logs issued lane addresses, not post-coalescer DRAM requests, and not L1/L2 hit/miss (use ncu for that). Physical channel interleave is invisible.
  • Batch = 1. Larger batch turns the decode GEMV into a GEMM; the streaming verdict only strengthens.
  • Paged KV (vLLM) — kernel, not full engine. The GPT-2/Qwen rows use HuggingFace's contiguous KV cache. The paged row is vLLM's real Triton paged-attention kernel (kernel_unified_attention) traced in isolation against a real paged KV pool with a scattered block table. We could not trace the full vLLM engine: NVBit deadlocks during vLLM's multi-stream / NCCL initialization (independent of attention backend), so the engine's weight GEMMs and KV-cache writes aren't included — only the genuine paged-decode read pattern.
