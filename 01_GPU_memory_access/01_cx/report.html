Summary

Measured LLM Weight and KV Cache Memory Access

I traced real CUDA memory accesses for GPT-2, GPT-OSS-20B, and Qwen2.5-1.5B, attributed addresses back to tensor ranges, and measured whether neighboring sectors and tiles are touched soon enough to justify large-block streaming decompression.

13.77GB GPT-OSS-20B CUDA weight ranges recorded, including MXFP4 custom tensors
1.35M GPT-OSS-20B KV-copy lane addresses attributed to cache tensors
0.977 mean 8KB tile coverage for GPT-OSS-20B KV-cache copy traffic
3 models GPT-2 baseline, GPT-OSS-20B, and Qwen2.5-1.5B HF/PyTorch traces

Verdict First

The traceable HF/PyTorch paths give a consistent cache conclusion: KV-cache update traffic is the strongest large-block candidate. Weight behavior depends on the kernel family: GPT-2 GEMV is stream-like at 1KB scale, while GPT-OSS dense projection weights are tiled/strided and need smaller sub-block handling.

Physical vs Temporal Locality

I treat "streaming" as two separate requirements. Physical nearby means traced addresses fill nearby sectors inside the same candidate tile, measured by tile coverage. Temporally adjacent means nearby lines appear soon after one another in trace order, measured by P(+N). A block is a good decompression target only when both hold at the same scale.

Why GPT-OSS Dense Weights Look Strided

The GPT-OSS and Qwen dense weights are not strided in tensor storage. The traced tensors are row-major contiguous BF16 matrices, but the projection kernels are CUTLASS/tensor-core GEMM kernels. For linear(x, W), the kernel consumes fragments of W as if reading W^T, so a warp can jump by an entire row stride while moving through a column-oriented tile.

Tensor Storage Layout Stride Jump Trace Effect
q/k/v_proj.weight Contiguous row-major, shape [out, 2880], stride [2880, 1] 2880 BF16 elements = 5760 bytes between adjacent rows Low 128B adjacency because lanes revisit matrix tiles rather than streaming one row linearly.
o_proj.weight Contiguous row-major, shape [2880, 4096], stride [4096, 1] 4096 BF16 elements = 8192 bytes between adjacent rows Large row-stride jumps make the memory trace appear tiled/strided even though storage is contiguous.

That is why the dense-weight trace has only 0.182 P(+1) at 128B and 0.193 mean 1KB tile coverage, while GPT-OSS KV copy reaches 1.000 mean 1KB coverage. KV copy is a contiguous data-movement kernel; dense projection GEMM is a tiled matrix kernel.

Limits and Next Experiments

This page reports only the targeted kernels that produced live, attributable NVBit records. Broader coverage should add longer prompts, larger models, multiple decode lengths, and more attention kernels so the block-size recommendation can be stress-tested across layouts and batch sizes.
