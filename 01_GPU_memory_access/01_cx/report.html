Summary

Measured LLM Weight and KV Cache Memory Access

I traced real CUDA memory accesses, attributed addresses back to tensor ranges, and measured whether neighboring sectors and tiles are touched soon enough to justify large-block streaming decompression.

5,000 weight-kernel NVBit records from the targeted GPT-2 GEMV grid
24,000 FMHA lane-level addresses attributed to KV-cache tensors
0.991 mean 4KB tile coverage for KV-cache FMHA reads
1KB / 8KB recommended starting blocks: weights / KV cache

Verdict First

The traceable HF/PyTorch path gives a clear answer: weight accesses are stream-like at cache-line and 1KB tile scale, while KV cache is the stronger large-block candidate. llama.cpp and TensorRT-LLM ran inference, but their production paths did not yield usable NVBit records in this environment.

Data Classification Strongest Evidence Compression Read
Weights streaming to 1KB P(+1) at 128B = 0.959; 1KB tile coverage mean = 0.767 Use 1KB blocks first; 4KB+ needs sub-block fallback.
KV cache streaming to 8KB FMHA P(+1) at 4KB = 0.968; 4KB tile coverage mean = 0.991 4KB-8KB streaming decompression is plausible.

TensorRT-LLM inference only Qwen Ray/no-MPI run passed and allocated paged KV cache. Ray worker emitted no MEMTRACE records; no access verdict claimed.

Methodology

Phase Implementation Output
Tensor range capture For HF/PyTorch, recorded CUDA tensor base address, end address, size, shape, stride, dtype, phase, and category. ranges.jsonl with 148 weight ranges and 48 KV-cache ranges.
Memory tracing Used the repo NVBit filtered memory trace tool with targeted grid IDs and kernel regex filters. trace.bin, launches.csv, and raw memtrace.txt.
Address attribution Mapped every nonzero traced address into recorded tensor ranges. Per-category CSVs such as weight_accesses.csv and kv_cache_accesses.csv.
Stream analysis Computed P(+N), consecutive runs, tile coverage, reuse distance, and stream-window neighbors. summary.csv and guide_metrics.json for each run.

Target Kernels

I did not trace every launch indiscriminately. The useful records came from kernels that directly model weight matrix-vector reads and KV cache update/attention reads.

Limits and Next Experiments

The production backends matter, but they need a different tracing strategy. llama.cpp under direct NVBit child preload stalled during CUDA/model initialization. TensorRT-LLM ran through Ray and emitted no MEMTRACE records from the worker. The next step is either a CUPTI activity path, an NVBit attach strategy that starts at selected kernels after context initialization, or backend-native instrumentation around KV page/block managers.
