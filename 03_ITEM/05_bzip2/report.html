bzip2: Burrows–Wheeler Compression

bzip2 takes a completely different route from gzip/zstd: instead of finding repeats with a sliding window, it reorders the data with the Burrows–Wheeler Transform so that similar characters cluster together, then rides that structure with a few light coders. Here is the whole pipeline, with a worked banana example you can follow by hand.

bzip2: RLE → BWT → MTF → RLE → Huffman (Julian Seward, 1996)  ·  block-based (≤900 KB)  ·  In our study: the best lossless codec on LLM weights.
The core trick. Most compressors look for repeated strings. bzip2 instead asks: “can I permute the bytes into an order that is trivially compressible, in a way I can perfectly undo?” The Burrows–Wheeler Transform does exactly that — it sorts rotations of the data so that characters appearing in similar contexts end up next to each other, producing long runs that cheap coders devour.

1. The pipeline (5 reversible stages)

input ─▶ RLE1 ─▶ BWT ─▶ MTF ─▶ RLE2 ─▶ Huffman ─▶ output (runs) (block (move- (runs (entropy sort) to-front) of 0s) code)
  1. RLE1 — an initial run-length pass (collapses long byte runs; a quirk that helps the BWT sorter).
  2. BWT — the heart: a reversible block sort that clusters similar characters.
  3. MTF (Move-To-Front) — turns those clusters into lots of small numbers, especially zeros.
  4. RLE2 — run-length-encodes the resulting runs of zeros.
  5. Huffman — entropy-codes the result (bzip2 actually picks among multiple Huffman tables per block).

Everything operates on one block at a time (100–900 KB, set by -1…-9).

2. The Burrows–Wheeler Transform, by hand

Take banana and append a unique end marker $ (smaller than every letter). List all rotations, then sort them; the BWT is the last column.

all rotations sorted rotations last column (BWT) banana$ $banana a anana$b a$banan n nana$ba sort ana$ban n ana$ban ─────▶ anana$b b na$bana banana$ $ a$banan na$bana a $banana nana$ba a BWT("banana$") = "annb$aa"

Notice the output annb$aa — the three as clustered, the two ns landed together. That is the magic: sorting rotations groups characters that precede similar following text, so the transformed string is full of runs even though the original was not. On real data (English, code, FP exponents) this clustering is dramatic.

It’s reversible. Astonishingly, the last column plus the marker is enough to reconstruct the original — the inverse BWT repeatedly sorts and follows a permutation (the “LF-mapping”). bzip2 stores the row index of the original string to anchor the inversion. No information is lost; it is a pure reordering.

3. Move-To-Front + RLE: turning clusters into zeros

MTF keeps an alphabet list; it outputs each symbol’s current position, then moves that symbol to the front. Runs of the same character therefore become runs of 0. Encoding the BWT output annb$aa with the alphabet list [$,a,b,n]:

readlist beforeoutputlist after
a$ a b n1a $ b n
na $ b n3n a $ b
nn a $ b0n a $ b
bn a $ b3b n a $
$b n a $3$ b n a
a$ b n a3a $ b n
aa $ b n0a $ b n

MTF output: 1 3 0 3 3 3 0 — small numbers dominated by zeros and repeats. A run-length pass (RLE2) then squeezes the zero runs, and Huffman assigns the shortest codes to the most frequent symbols (here, 0 and 3). The original, near-flat byte distribution has been turned into something highly skewed — exactly what entropy coding wants.

4. Compress / decompress in pseudocode

function bzip2_compress(data):
    for each block of up to 900 KB:
        b = rle1(block)                       # collapse long runs
        (bwt, idx) = burrows_wheeler(b)       # sort rotations; idx = original row
        m = move_to_front(bwt)                # clusters -> small ints / zeros
        r = rle2(m)                           # run-length the zeros
        emit(idx, huffman_encode(r))          # entropy code (multiple tables)

function bzip2_decompress(stream):           # exact inverse, in reverse order
    for each block:
        r   = huffman_decode(...)
        m   = un_rle2(r)
        bwt = inverse_mtf(m)
        b   = inverse_bwt(bwt, idx)           # LF-mapping reconstruction
        emit(un_rle1(b))

The cost is in the BWT’s suffix sort (sorting all rotations of the block) — that is what makes bzip2 slower than gzip but its decompression is straightforward and fast.

5. Characteristics & when to use it

  • Block-based, no cross-block window. The BWT works within one ≤900 KB block; bzip2 can’t exploit repetition farther apart than that — unlike xz’s huge dictionary.
  • Strong on locally-structured data — text, source code, and (in our study) floating-point weight bytes, where the exponent/sign bytes cluster beautifully under the BWT.
  • Slower compress, fast decompress; moderate memory; a single, dependency-free, widely-available CLI.
  • Weaker on long-range repetition (large files with distant duplicates) — there a big-window LZ coder like xz or zstd wins.
In our LLM study, bzip2 was the best lossless codec on weights (~1.44× on bf16, 2.54× on fp32) — its Burrows–Wheeler block sort suits the structured exponent bytes of floating-point parameters better than any sliding-window coder. On KV cache, whose gains come from long-range repetition, the large-window xz overtook it.
