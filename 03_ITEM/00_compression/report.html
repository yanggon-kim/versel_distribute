Producer-Consumer Compression on Memory-Bound GPU Kernels

Can a compression stage run in the idle compute cycles of a memory-bandwidth-bound kernel to cut traffic and improve end-to-end latency? Four staged studies, profiled on an RTX 5080.

GPU: NVIDIA GeForce RTX 5080 (Blackwell, sm_120, 16 GB, ~960 GB/s)  ·  Toolkit: CUDA 13.2  ·  Profiler: Nsight Compute 2026.1  ·  32 kernels across 4 families, validated vs CPU reference.
Founding question. A memory-bound kernel spends most of its time stalled on memory, leaving the SM's arithmetic units idle. Can compression run in those idle cycles — reducing traffic — without slowing the producer? The goal was evidence, not confirmation: a "NO" is as valid a result as a "YES".

The four studies at a glance

Each study aims compression at a different traffic stream. The arc: useless on minority traffic, near-linear on the dominant read, real-but-throttled on the dominant write, and about half of that again once real ops and a real 4× codec are used.

#StudyWhat was compressedAnswerBest speedup
1Output (V0–V7) the output of a read-bound decode kernel (≈0.05% of traffic) NOnone (flat)
2Read / KV (R0–R7) the dominant read (compressed KV cache) YES~7.8× @ 8×
3Write (W0–W7) the dominant write (6 real-workload profiles + transformer) YES, sub-linear~2.6–3.1×
4Real kernels real ops + authentic int8 codec (4×) YES, smaller1.2–1.7×

Study 1 — Output compression (read-bound decode): NO

A decode-attention-like kernel was confirmed strongly memory-bound — DRAM 96%, SM 6.5%, long-scoreboard stalls 65% — with ~1000 dependent FLOPs/output that hide for free (<1% latency). Yet compressing the output moved total DRAM bytes <0.1% (output is ~0.05% of traffic), so runtime was flat across every compression ratio. The lesson that frames the whole series: free compute ≠ benefit — you must compress the dominant traffic.

Study 2 — Read / KV compression: YES (≈ linear)

Storing the KV cache as sequence block-means (an exact sufficient statistic for a mean reduction, so still bitwise-valid) shrank the dominant read by the compression ratio. Across all 8 structures, end-to-end time tracked traffic almost perfectly:

This is the win Study 1 could not get — because the lever was on the traffic that dominates.

Study 3 — Write-side compression: YES, sub-linear

A write-dominant producer (read small X, write a large compressed output). Six write-heavy workloads were found via literature search and run as traffic-profile kernels, plus a realistic transformer activation-save. Best-variant speedups:

Sub-linear (vs Study 2's linear) because pure store streams under-saturate HBM (~54% of peak vs ~95% for reads) and efficiency falls further as the write shrinks. New structure result: the two-kernel variant is catastrophic on the write side (it writes the full uncompressed output, then re-reads it), while TMA-store is best. Fusion is mandatory.

Overlap evidence — compression really runs in memory-stall idle cycles

On the memory-bound kernel, injected codec FLOPs are free up to a knee: AdamW ~16–24 FLOPs/element (load-latency-bound), transformer / KV ~8, stencil ~4. Two limits the data pinned down: the idle budget is bounded (heavy codecs exceed it → no benefit), and it only exists if the kernel is memory-bound — a naive issue-bound kernel had ~0 free FLOPs. Make it memory-bound first; then the compression is free.

Study 4 — Real kernels with authentic int8 codecs: YES, ~half of synthetic

Each candidate was rebuilt with its real op (GeLU erff, RoPE sin/cos, 7-point stencil, Adam update, routed scatter) and a real int8 codec (4×, validated to ≤0.5 quantization step, 0 sanitizer errors):

Nsight Compute shows the cause directly: int8 flips kernels from memory-bound to compute-bound (activation DRAM 85→24%, SM 58→69%; transformer DRAM 92→35%, SM 50→87%). The real op's arithmetic doesn't shrink, so it re-consumes the idle that compression freed — exactly the bounded-idle-budget prediction. The synthetic proxy (1 FMA) over-stated the win by staying memory-bound; MoE's cheap copy op is the exception that lands near the int8 traffic ideal of 1.6×.

The eight implementation variants, in detail

Every study runs the same eight structures (numbered 0–7; prefixed V for output, R for read, W for write). They all compute the same result and differ only in how the compression codec is attached to the memory work — which is exactly what determines performance on a memory-bound kernel.

#VariantHow it worksVerdict
0Baseline One thread per output element; read input, compute, compress, store — the simplest mapping and the reference point. No staging, no roles. solid, simple
1Same-CTA (fused) One thread block per row: cooperatively load the input into shared memory, then every thread computes → compresses → stores inside the one CTA. The codec rides with the issuing threads. top tier
2Warp-specialized Block split into producer warps (compute values into a shared tile) and consumer warps (read tile, compress, store), phased with barriers — the classic producer/consumer role split. poor (memory-bound)
3Two-kernel Kernel A writes the full uncompressed intermediate to DRAM; kernel B reads it back and compresses. A control that separates "compression benefit" from "overlap benefit" — with no overlap and an extra round-trip. worst on writes
4Persistent Launch ~one block per SM; blocks stay resident and grid-stride over the rows, fusing compute+compress+store like #1 but amortizing launch and keeping pipelines warm. within 1–2% of best
5cp.async pipeline Double-buffered asynchronous copies (cp.async) stage inputs into shared memory so the load of the next tile overlaps the compute of the current one (SM80+). top tier
6Cluster + DSM A 2-CTA thread-block cluster: the producer CTA computes into its shared memory; the consumer CTA reads it across Distributed Shared Memory and stores (Hopper+). A CTA-level producer/consumer split. poor
7TMA The Tensor Memory Accelerator issues bulk asynchronous copies under an mbarrier — TMA loads for reads, TMA stores for writes — the closest match to the modern hardware data-movement path. best overall

The split-role variants (#2 warp-specialized, #6 cluster/DSM) and the unfused #3 two-kernel consistently lose, because they move the codec off the threads that issue the memory traffic — a memory-bound kernel can't spare that issue parallelism (its consumer warps/CTAs sit idle of memory work). Warp specialization is a win for compute-bound Tensor-Core pipelines (GEMM, FlashAttention), not for the bandwidth-bound reductions/copies here.

Below, each variant is shown as simplified pseudo-CUDA (write-side framing; read/output are analogous). Two helpers appear throughout: produce() = the real op (GeLU, reduce, RoPE…), and compress() = the codec (quantize / group-sum) — both run on the CUDA cores, i.e. in the idle compute the memory stalls create.

#0 — Baseline thread-per-element

The simplest mapping and the reference point: one thread owns one output element and does the whole pipeline itself.

// grid covers all N*Dc outputs, one thread each
i = blockIdx.x * blockDim.x + threadIdx.x;
n = i / Dc;   c = i % Dc;
v       = produce(X, n, c);     // real compute, in this thread
out[i]  = compress(v);          // codec, in this thread

Result: solid and simple; near the memory-bound floor.

#1 — Same-CTA (fused) one block per row

The block stages the row's input in shared memory, then every thread computes → compresses → stores inside that one CTA. The codec rides with the issuing threads.

__shared__ float xs[Din];
n = blockIdx.x;
cooperatively load X[n][0..Din) into xs;
__syncthreads();
for (c = threadIdx.x; c < Dc; c += blockDim.x)
    out[n*Dc + c] = compress(produce(xs, c));   // compute + compress + store, fused

Result: top tier — the simplest fused design that wins.

#2 — Warp-specialized producer / consumer warps

The block is split into producer warps (compute values into a shared tile) and consumer warps (read the tile, compress, store), handed off through shared memory.

__shared__ float tile[ROWS * Dout];
warp = threadIdx.x / 32;
if (warp < NPROD)                        // PRODUCER warps: compute into shared tile
    for (idx in this warp's range) tile[idx] = produce(X, idx);
__syncthreads();                          // hand-off through shared memory
if (warp >= NPROD)                       // CONSUMER warps: compress + store
    for (idx in this warp's range) out[idx] = compress(tile[idx]);

Result: poor on memory-bound kernels — the consumer warps issue no memory, so achieved bandwidth drops (read study R2 = 0.44×).

#3 — Two-kernel unfused control

Kernel A writes the full uncompressed intermediate to DRAM; kernel B reads it back and compresses. Separates "compression benefit" from "overlap benefit" — at the cost of a full extra round-trip.

// kernel A: produce the FULL uncompressed result to DRAM
Y[i] = produce(X, i);
// kernel boundary: all of Y round-trips through HBM
// kernel B: read Y back, compress, store
out[j] = compress(combine(Y, j));

Result: worst on writes (writes full-size data first) — up to 6.9× slower than TMA.

#4 — Persistent grid = #SMs, grid-stride

Launch ~one block per SM; blocks stay resident and walk the rows with a grid-stride loop, fused like #1 but amortizing launch and keeping pipelines warm.

// gridDim = numSMs * blocksPerSM  (blocks stay resident)
for (n = blockIdx.x; n < N; n += gridDim.x) {     // grid-stride over rows
    load X[n] into shared xs;  __syncthreads();
    for (c = threadIdx.x; c < Dc; c += blockDim.x)
        out[n*Dc + c] = compress(produce(xs, c));
}

Result: within 1–2% of the leaders.

#5 — cp.async pipeline double-buffered async load

Asynchronous copies stage the next input tile while the current one is being computed, so the load latency hides behind the compute (SM80+).

__shared__ float buf[2][TILE];                  // double buffer
cp_async(buf[0] = X tile 0);  commit();
for (t = 0; t < ntiles; t++) {
    if (t+1 < ntiles) { cp_async(buf[(t+1)&1] = X tile t+1); commit(); }  // prefetch next
    wait_prior(1);  __syncthreads();            // overlap: next load with this compute
    for (e in buf[t&1]) out[...] = compress(produce(e));
}

Result: top tier — full bandwidth at lower warp occupancy.

#6 — Cluster + DSM 2-CTA cluster, Distributed Shared Memory

A 2-CTA thread-block cluster: the producer CTA computes into its own shared memory; the consumer CTA reads it across Distributed Shared Memory and stores (Hopper+).

cluster = this_cluster();  rank = cluster.block_rank();   // 2 CTAs / cluster
__shared__ float buf[ROWS * Dout];
if (rank == 0)                             // PRODUCER CTA: compute into its own smem
    for (idx) buf[idx] = produce(X, idx);
cluster.sync();
if (rank == 1) {                           // CONSUMER CTA: read peer smem via DSM
    float* peer = cluster.map_shared_rank(buf, 0);   // Distributed Shared Memory
    for (idx) out[idx] = compress(peer[idx]);        // then store
}

Result: poor — the consumer CTA drives no DRAM reads/writes of its own.

#7 — TMA cores compress, hardware engine moves

The key point: TMA does not compress. The CUDA cores compute and compress into a shared-memory tile; the Tensor Memory Accelerator (a bulk async copy engine) then ships the already-compressed tile to DRAM under an mbarrier. "TMA variant" names the data-movement path, not the codec. 

__shared__ alignas(128) int8_t tile[CW];        // compressed staging
n = blockIdx.x;
for (c0 = 0; c0 < Dout; c0 += CW) {
    if (threadIdx.x < CW)
        tile[threadIdx.x] = compress(produce(X, n, c0 + threadIdx.x));  // CORES compress into smem
    __syncthreads();
    if (threadIdx.x == 0)
        cp_async_bulk_tensor_2d_s2g(tmap, c0, n, tile);  // TMA moves compressed bytes to DRAM
    cp_async_bulk_wait();
    __syncthreads();
}
// host, once:  cuTensorMapEncodeTiled(&tmap, out, shape, box);  // WHERE to copy; no compression
Why TMA wins even though it doesn't compress: the cores already shrank the bytes (e.g. int8, 4×); TMA then moves those fewer bytes at ~80% of peak DRAM (vs ~54% for naive per-thread stores) and frees the cores from store-address generation, leaving them to keep computing/compressing. The read variant (R7) is the mirror: TMA loads compressed tiles into shared memory, then the cores decompress + reduce.

Result: best overall, especially for writes (TMA-store).

The unifying law

Compression speeds up a memory-bound kernel only to the extent it shrinks the dominant traffic, only when the codec's compute fits inside the idle the memory stalls create, and only when it rides inside the threads that issue that traffic.

Each clause is a measured failure mode:

  • Dominant traffic — compressing 0.05% of traffic did nothing (Study 1); the dominant read/write worked (Studies 2–3).
  • Codec ≤ idle budget — heavy codecs and real GeLU/RoPE exceed the ~4–24 free FLOPs/element and flip the kernel compute-bound (Studies 3–4).
  • Ride inside the issuing threads — the producer/consumer split and the unfused two-kernel always underperform; fused same-CTA / cp.async / TMA win (every study).

Two hard ceilings recur: the read floor (you cannot compress the uncompressible input — caps AdamW at ~1.3×) and store under-saturation (pure-write kernels peak at ~55% of HBM, so write wins are sub-linear).

Practitioner guidance

  • Compress the dominant traffic. For decode attention that is the KV read (near-linear); output compression is pointless.
  • Use a cheap codec (int8/int4 absmax, block-mean). Heavy codecs (top-k, low-rank) pay off for network bandwidth, not on-GPU HBM time.
  • Fuse the codec into the producer. Never a separate compression kernel; never a producer/consumer warp/CTA split. Prefer TMA-store / cp.async / same-CTA.
  • Expect ~1.2–1.7× on real training writes (activation save, optimizer states, MoE) from a real 4× codec — modest but real, and it stacks with the memory-capacity savings these techniques actually target.
  • Watch for the compute-bound flip. Arithmetic-heavy ops (GeLU, softmax, trig) go compute-bound quickly once compressed; profile DRAM% before and after.

Related work

"Idle compute does compression to relieve the memory wall" is a real, named research line. The closest prior art is CABA; this project is a modern, software-only, profiled re-examination of that thesis on Blackwell, with a per-structure breakdown and a real-vs-synthetic calibration.

Bullseye — idle GPU compute for bandwidth compression

  • CABA: Core-Assisted Bottleneck Acceleration (Vijaykumar, Jog, Mutlu et al., ISCA 2015) · extended framework (arXiv 1602.01348) — uses idle GPU compute via "assist warps" to compress/decompress data and ease the memory-bandwidth bottleneck (+41.7% avg, up to 2.6×). Essentially this study's hypothesis, in hardware.
  • Morpheus (MICRO 2022) — puts idle cores into a "cache mode" helper kernel that extends the LLC with cache compression (+39% on memory-bound workloads). Same spirit, capacity rather than write traffic.

The shipping realization — fused dequantization for LLMs

  • FLUTE, Atom (MLSys 2024), CodeGEMM, Multi-Scale Dequant, ZipServ — fetch compressed weights from HBM and decompress on-chip in the idle compute, feeding tensor cores with no HBM round-trip; speedup is linear in the ratio in the memory-bound regime (this study's read-side result, productized).

Architectural memory / bandwidth compression

  • Base-Delta-Immediate (PACT 2012), Toggle-Aware Compression (ASPLOS 2015), memory/IO link compression for GPGPU — cheap codecs and on-the-fly (de)compression to multiply effective bandwidth/capacity (plus the energy/bit-toggle caveat); the hardware analogue of "cheap codec, fused."

GPU decompression engines & overlap

  • CODAG (2023), NVIDIA nvCOMP / RAPIDS — general GPU decompression that overlaps decode with memory latency (prefetch warp stages chunks while another decodes) — the engineering of "ride inside the issuing threads."

Workloads benchmarked here (data-compression counterparts)

  • ActNN / GACT / CompAct (activations), 8-bit Adam (optimizer states), KVQuant / MLA (KV cache), DGC / PowerSGD (gradients), SZ / ZFP (HPC checkpoints) — cited in the sub-reports; this study tests them under the idle-compute-overlap lens.

How this study differs

CABA owns the core thesis and fused-dequant ships it, but three angles appear under-covered: (1) a systematic per-structure comparison of how to attach the codec (eight variants), with the clean result that the producer/consumer split and the unfused two-kernel hurt while fused / TMA win; (2) the read-vs-write asymmetry — read compression is near-linear, write compression sub-linear because store streams under-saturate HBM (~55% vs ~95%); (3) a measured free-FLOP budget per workload and the synthetic→real gap, where real GeLU/RoPE flip the kernel compute-bound and roughly halve the predicted win.

Final answer

Yes — producer-consumer compression can exploit a memory-bound kernel's idle compute, but the benefit is governed by three gates and two ceilings. It is near-linear aimed at the dominant read (KV-cache decode), real but sub-linear for dominant writes (~1.3–1.7× with a real int8 codec on this GPU), bounded by the read floor and store-bandwidth ceiling, and nullified by heavy codecs or by splitting the producer from the consumer. Aimed correctly — dominant traffic, cheap codec, fused — the original hypothesis holds; aimed carelessly, it buys nothing. Both outcomes are backed by measurement.
