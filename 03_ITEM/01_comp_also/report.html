LZSS: From Concept to Implementation, and GPULZ

A ground-up explanation of the LZSS lossless compression algorithm — the sliding window, the encode/decode loops in plain pseudocode, and what makes it tick — followed by how GPULZ reshapes that inherently sequential algorithm to run in parallel on a modern GPU.

Topic: dictionary compression (LZ77 family)  ·  Scope: concepts → pseudocode → characteristics → GPU parallelization  ·  Measurements: Canterbury corpus + GPULZ on an RTX 5080 (CUDA 12.8, sm_120).
The one-sentence idea. Text and data repeat themselves; instead of storing a repeated run of bytes again, store a short pointer back to where it already appeared. LZSS is the careful, practical way of doing exactly that — and of only doing it when the pointer is actually smaller than the bytes it replaces.

1. Where LZSS comes from: the LZ family

LZSS is a member of the LZ77 family of dictionary (or substitutional) compressors. The shared idea, introduced by Abraham Lempel and Jacob Ziv in 1977, is that the text already seen is the dictionary: any sequence about to be written can be replaced by a reference to an earlier, identical sequence.

  • LZ77 (1977) emits a stream of triples (offset, length, next_symbol). Every token — even one describing a match — carries one explicit literal symbol at the end. That trailing literal is pure overhead whenever a match is found.
  • LZSS (Storer & Szymanski, 1982) fixes that waste with two changes:
    1. A single flag bit per token says “literal” or “match”, so a match no longer has to drag a literal along.
    2. A match is emitted only if it pays for itself (the breakeven rule below). Short, unprofitable matches are written as plain literals instead.

Those two tweaks are the whole difference, and they are why essentially every practical “LZ77” codec in the wild — the match-finder inside DEFLATE (gzip, zlib, PNG, ZIP), LZ4, and many others — is really LZSS-style.

2. The sliding window

LZSS scans the input left to right with a fixed-size window split into two parts around the current cursor:

  • Search buffer (the dictionary): the last D symbols already encoded, sitting behind the cursor. Matches are searched for here.
  • Look-ahead buffer: the next L symbols ahead of the cursor, waiting to be encoded. The longest prefix of this buffer that also appears in the search buffer is the match.
already encoded . . . . . . . . . . | . . . not yet encoded +-----------------------------------+---+-------------------------+ | a b r a c a d a b r a | a | b r a c ... | +-----------------------------------+---+-------------------------+ <------- search buffer (size D) -----> ^ <-- look-ahead (size L) --> cursor match for "abra" found 7 symbols back -> emit (offset=7, length=4)

Both parts move together as the cursor advances — hence “sliding”. The window is the algorithm’s entire memory: nothing older than D symbols can be referenced. D and L are the two knobs that define a given LZSS configuration.

3. Token format and the breakeven rule

The compressed stream is a sequence of tokens, each prefixed by its flag bit:

0 + literal symbol (e.g. 8 bits)     1 + offset (⌈log₂D⌉ bits) + length (⌈log₂L⌉ bits)

A literal costs 1 + b bits (where b is the symbol width). A match costs 1 + log₂D + log₂L bits but stands in for length symbols. So a match is only worthwhile when it replaces more bits than it costs. The minimum profitable match length is the breakeven length:

Breakeven rule. Encode a match only if length × b > log₂D + log₂L, i.e. MIN_MATCH = ⌊(log₂D + log₂L) / b⌋ + 1. Anything shorter is cheaper to write as literals.

For a typical byte-oriented setup (b = 8, D = 4096 ⇒ 12-bit offset, L = 16 ⇒ 4-bit length) the pointer is 16 bits, so MIN_MATCH = 3: matches of length 1–2 are written as literals, matches of 3+ as pointers. This single rule is what keeps LZSS from ever inflating data badly — the worst case is one extra flag bit per literal.

4. Encoding (compression) — the concept in pseudocode

The encoder repeatedly finds the longest match for the look-ahead buffer inside the search buffer, then either emits a pointer (if it beats breakeven) or a single literal, and slides the window forward by however many symbols it just consumed. 

# Inputs: data[0..N-1], window sizes D (dictionary) and L (look-ahead)
function LZSS_encode(data, D, L):
    pos = 0
    while pos < N:
        # search the dictionary [pos-D, pos) for the longest prefix of data[pos..]
        (best_off, best_len) = find_longest_match(data, pos, D, L)

        if best_len >= MIN_MATCH:
            emit_bit(1)                       # flag: this is a match
            emit_bits(best_off, ceil(log2(D)))
            emit_bits(best_len, ceil(log2(L)))
            pos += best_len                   # consume the whole match
        else:
            emit_bit(0)                       # flag: this is a literal
            emit_bits(data[pos], b)           # write one raw symbol
            pos += 1                          # consume one symbol

function find_longest_match(data, pos, D, L):
    start = max(0, pos - D)
    best_off = 0; best_len = 0
    for j in start .. pos-1:                  # every candidate start in the dictionary
        len = 0
        while len < L and pos+len < N and data[j+len] == data[pos+len]:
            len += 1                          # extend the match (may overlap into look-ahead)
        if len > best_len:
            best_len = len; best_off = pos - j
    return (best_off, best_len)
Where all the cost lives. find_longest_match is the expensive part. The naive version above is O(N × D × L). Real encoders accelerate the search with hash chains (zlib), binary search trees, or suffix automata, and many use lazy matching — before committing a match at pos, check whether starting one symbol later yields a longer one — to squeeze out a few extra percent of ratio. The decode side is untouched by any of this.

5. Decoding (decompression) — trivially fast

Decoding needs no search at all. It reads the flag, then either copies one literal through, or copies length already-decoded symbols from offset positions back. The output buffer is the dictionary, so it reconstructs itself as it goes. 

function LZSS_decode(stream):
    out = []
    while not stream.eof():
        if read_bit() == 0:                    # literal
            out.append(read_bits(b))
        else:                                 # match: copy from earlier output
            off = read_bits(ceil(log2(D)))
            len = read_bits(ceil(log2(L)))
            start = length(out) - off
            for k in 0 .. len-1:
                out.append(out[start + k])    # byte-by-byte: handles overlap (off < len)
    return out
Overlapping copies are a feature. When offset < length the copy reads symbols it is still writing — copying one symbol at a time turns this into a run-length fill. That is how LZSS encodes a long run like aaaaaaaa as a single cheap match (offset=1, length=7) after the first a.

6. A tiny worked example

Encoding abracadabraabracadabra (showing the token stream conceptually):

abr aca d (off 7, len 4) → "abra" (off 11, len 11) → "abracadabra"

The first seven symbols have nothing behind them long enough to beat breakeven, so they go out as literals. By the time the cursor reaches the second abra, a 4-symbol match exists 7 back; and the entire second half abracadabra is one 11-symbol match pointing 11 back — the overlap-into-look-ahead behavior lets a single pointer cover a long repeat.

7. Characteristics of LZSS

PropertyWhat it means in practice
LosslessOutput decodes bit-for-bit to the original. Used wherever exact reconstruction is mandatory.
Asymmetric costSlow to encode (the match search dominates), very fast to decode (just memory copies). Ideal for “compress once, decompress many” like software distribution and assets.
Single-pass, adaptiveNo statistics or training needed; the dictionary is built on the fly from the data itself. It can compress a stream it has never seen, with no header model.
Window-size tradeoffA larger dictionary D finds more/longer matches (better ratio) but costs more search time and needs more offset bits per pointer. D and L tune ratio vs speed.
Bounded expansionThanks to the breakeven rule, incompressible data grows by only ~1 flag bit per symbol — it never blows up.
Parsing quality mattersGreedy longest-match is simple but sub-optimal; lazy or optimal parsing trades encode time for a better ratio.
Inherently sequentialEach token depends on the dictionary built by all prior tokens. This data dependency is exactly what makes LZSS hard to parallelize — the central problem GPULZ tackles.
A building blockLZSS produces (literal/match) tokens; an entropy coder (Huffman, arithmetic, ANS) then packs them. DEFLATE = LZSS + Huffman.

Measured on the full Canterbury corpus (a CPU LZSS, dictionary-style), the compressed output averaged roughly 40% of the original size on mixed text and binary — typical for a standalone LZSS without a following entropy stage.

8. GPULZ — running LZSS on a GPU

GPULZ (Zhang et al., ICS’23) is an LZSS compressor designed for multi-byte data on modern GPUs. The motivation: HPC and scientific applications generate huge volumes of multi-byte data (int32, float32), and compression cuts storage and inter-node communication cost — but classic LZSS is sequential and single-byte oriented, so prior GPU LZSS implementations had poor throughput.

8.1 The two obstacles, and GPULZ’s answers

Obstacle in classic LZSSGPULZ’s approach
The sliding window is globally sequential — token i depends on all tokens before it. Block independence. Split the input into fixed-size blocks (e.g. BLOCK_SIZE = 2048 bytes). Each block is compressed independently by one GPU thread block, with its own private window. Thousands of blocks run in parallel.
Single-byte symbols mean a huge number of positions and short matches, especially on numeric data. Multi-byte symbols. Treat the unit as a multi-byte type (INPUT_TYPE = uint32_t), so the window size is measured in elements, not bytes. Fewer positions to match (lower complexity) and longer repeated patterns are found in int/float data.
Variable-length output: each position emits a literal or a pointer, so you don’t know where each thread’s bytes go until you know everyone’s lengths. Prefix sum + kernel fusion. A parallel scan computes each token’s output offset for compaction. GPULZ restructures this from one pass into two passes and fuses kernels to cut data movement between shared and global memory.

8.2 Within a block: parallel matching, then compaction

Inside one block, every position is matched against its window at the same time by a thread. The per-position results then drive a flag array (1 bit per token: match vs literal) and a prefix sum that lays out the variable-length encoded stream. 

# Conceptual GPULZ block compression (one thread block per data block)
parallel for each position i in block:          # one thread per element
    window = block[max(0, i-WINDOW_SIZE) .. i-1]   # WINDOW_SIZE in units of INPUT_TYPE
    (off, len) = longest_match(block, i, window)   # multi-byte symbol compare
    if len >= MIN_MATCH:
        flag[i] = 1;  token[i] = (off, len);  cost[i] = pointer_size
    else:
        flag[i] = 0;  token[i] = block[i];    cost[i] = symbol_size

barrier()                                          # whole block in shared memory

# Pass 1 + Pass 2 prefix sum over cost[] -> output position of every token
offset[] = exclusive_prefix_sum(cost[])            # parallel scan, two-pass + fused

parallel for each emitted token t:               # scatter, no collisions
    write flag bit into block's flag region
    write token[t] at offset[t] in block's data region

# Per block, store: (flag array) + (compacted token stream).
# A block-offset table records where each block lands in the global output,
# so decompression can find and expand all blocks in parallel too.

Decompression mirrors this: each block is independent, so a thread block reads its flag array, and threads expand literals and copy matches into the block’s output region in parallel. Because blocks never reference each other, both directions scale with the number of blocks.

8.3 Architectural tuning and the ratio tradeoff

  • Maximize shared memory. GPULZ sizes its data partitions so each block’s working set (input, window, flags, scan buffers) fits in fast on-chip shared memory, adapting the partition to the target GPU architecture.
  • Compile-time configuration. BLOCK_SIZE, WINDOW_SIZE (≤ 255), and INPUT_TYPE are #defines — the data type is fixed when the kernel is built, because C++ has no runtime type selection.
  • Ratio cost of independence. Per-block windows can’t reference matches outside their block, which slightly lowers ratio versus a single global window — the price paid to unlock massive parallelism.
Reported results (paper). Up to 272.1× compression throughput speedup on an NVIDIA A4000 and up to 1.4× higher compression ratio than the state-of-the-art GPU LZSS, evaluated on multi-byte scientific datasets across A100 and A4000 GPUs.

Why text barely compresses under GPULZ. Its symbol is a 4-byte uint32_t, so two text fragments only “match” when four consecutive bytes align identically — rare in prose. On numeric data with repeated 4-byte values it shines (11.5× here); on byte streams a byte-oriented CPU LZSS (~40% of original on Canterbury) wins. The lesson: GPULZ is purpose-built for multi-byte scientific data, not general text.

9. Summary

LZSS is LZ77 made practical: a sliding window, a flag bit to tell literals from back-references, and a breakeven rule so a pointer is only used when it actually saves space. Encoding is a search problem (slow, lots of tuning room); decoding is just guided copying (fast). Its one structural weakness — the sequential window dependency — is precisely what GPULZ engineers around: independent blocks for parallelism, multi-byte symbols for numeric data, and a fused two-pass prefix sum to lay out variable-length output on the GPU. The result is enormous throughput on the multi-byte scientific data it targets, at the cost of being a poor fit for ordinary text.
