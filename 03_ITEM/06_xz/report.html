xz / LZMA

xz is the container; LZMA is the algorithm inside. It is, in spirit, an LZ77 dictionary coder like LZSS — but with a huge dictionary and a much smarter back end: a range coder driven by hundreds of tiny adaptive probability models. That combination is why xz usually gives the highest ratio of the everyday codecs.

xz: LZMA2 + integrity check (Tukaani, 2009)  ·  LZMA: large-window LZ77 + context-modelled range coder  ·  In our study: the best lossless codec on LLM KV cache.
Two ideas stacked. (1) Find repeats with a very large window (up to 64 MB–1.5 GB, vs gzip’s 32 KB), so distant duplicates still compress. (2) Encode the resulting literals and matches with a range coder that predicts each bit using a context-specific probability it constantly updates — squeezing far closer to the entropy limit than gzip’s whole-bit Huffman codes.

1. The two stages

raw bytes ─▶ [ big-window LZ match finder ] ─▶ packets (literal / match / rep-match) up to 64 MB dictionary ─▶ [ range coder w/ adaptive bit-probability models ] ─▶ compressed stream
  • Match finding — like LZSS, replace repeats with (distance, length), but search a dictionary thousands of times larger, and (at high presets) with an optimal parse that picks the cheapest sequence of packets, not just the greedy longest match.
  • Range coding — an arithmetic-coding variant. Instead of fixed codes, every decision (is this a literal or a match? what is this bit of the literal?) is coded against an adaptive probability chosen by context, getting fractional-bit efficiency.

2. LZMA packets — and why “repeat distances” matter

LZMA emits one of a few packet types per step. Crucially, it caches the 4 most recent match distances, so re-using a recent distance is extremely cheap — ideal for strided or columnar data.

packetmeaningcost
LITERALone raw byte (predicted from previous byte + position)~ a few bits
MATCHnew (distance, length) back-referencefull distance + length
REP0–REP3reuse one of the 4 most-recent distances, new lengthcheap (no new distance)
SHORTREPone byte at the last distancecheapest

3. A small example

Encoding abracadabraabracadabra conceptually (LZ stage):

abr aca d MATCH dist 7, len 4 → "abra" MATCH dist 11, len 11 → "abracadabra"

The LZ packets look just like LZSS. The difference is the back end: each literal byte is range-coded using a probability model keyed on the previous byte (so after a, the model already expects b/c/d and spends well under 8 bits); each match is coded with separate context models for its length and the slots of its distance. On data with regular structure, the REP distances turn repeated strides into near-free packets. The range coder then packs the whole packet stream into a single fractional-bit number.

Range coding in one line. Keep an interval [low, high); to encode a symbol of probability p, shrink the interval to the sub-range of width p; emit bytes as the top of the interval settles. More probable symbol ⇒ wider sub-range ⇒ fewer bits. The probabilities are adaptive — updated after every bit — so the model learns the data as it goes.

4. Pseudocode

function lzma_compress(data, dict_size):           # dict up to 64 MB+
    reps = [1,1,1,1]                                  # 4 recent distances
    init probability models (literal, isMatch, isRep, len, dist...)
    pos = 0
    while pos < len(data):
        (dist, length) = best_match(data, pos, dict_size)   # optimal parse at high presets
        if length < MIN or literal_is_cheaper:
            range_encode_bit(isMatch[ctx], 0)
            range_encode_literal(prev_byte, pos, data[pos]); pos += 1
        else:
            range_encode_bit(isMatch[ctx], 1)
            if dist in reps:
                range_encode_rep(index_of(dist, reps), length)   # REP0..3 — cheap
            else:
                range_encode_match(dist, length); push dist into reps
            pos += length
        update all touched probability models                # adaptive

# Decompression mirrors this: range-decode each decision, copy literals/matches
# from the window. It is fast and uses little memory beyond the dictionary.

5. Characteristics & when to use it

  • Highest ratio of the common codecs, especially on large inputs with long-range repetition — the huge dictionary sees duplicates gzip/bzip2 can’t reach.
  • Slow to compress (large search + optimal parse) and memory-hungry (the dictionary); decompression is fast and light.
  • Adaptive context models + REP distances make it strong on structured / strided data (tables, columns, repeated records).
  • xz container adds framing + an integrity check (CRC/SHA) around the LZMA2 stream; the -1…-9 presets mainly trade dictionary size and search effort.
In our LLM study, xz was the best lossless codec on KV cache (~2.6–3.3× on bf16, 4.87× on fp32) — KV compressibility comes from long-range repetition, and xz’s dictionary (up to 64 MB) reaches it where smaller-window coders cannot. On weights, which are entropy-bound rather than repetitive, the Burrows–Wheeler bzip2 edged it out.

6. xz vs the others, in one line each

  • vs gzip: same LZ+entropy shape, but 2000× larger window and a range coder instead of Huffman → much higher ratio, much slower.
  • vs zstd: xz usually compresses a bit smaller at max settings; zstd decompresses faster and is far quicker to compress — zstd wins the speed/ratio trade-off, xz wins pure ratio.
  • vs bzip2: different families — xz is big-window LZ (great on long-range repeats / KV cache); bzip2 is block-sort BWT (great on locally-structured data / weights).
