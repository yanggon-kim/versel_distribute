Compressing LLM Weights & KV Cache

A fair, verified compression-ratio study of three memory-oriented compressors — LZSS, GPULZ, and BPC — against gzip/zstd baselines, on real weights and KV-cache tensors from gpt-oss-20b, Qwen2.5-3B, and Mistral-7B.

Models: gpt-oss-20b (MXFP4), Qwen2.5-3B, Mistral-7B-Instruct (bf16)  ·  Data: weights + KV cache, in fp32 and native bf16  ·  546 measurements, all round-trip / self-verified  ·  Host: RTX 5080, CUDA 12.8, torch 2.9.1, transformers 4.57.1.
The question. How well do GPU/memory compressors built for other data — LZSS (byte dictionary), GPULZ (GPU LZSS on 4-byte symbols), BPC (bit-plane transform) — compress actual LLM tensors, and how do they stack up against everyday gzip/zstd? All ratios are original ÷ compressed (higher = better), reported as the geometric mean.

1. Setup

  • Tensors (layers {0, mid, last}, deterministic): attention q_proj/o_proj, MLP down_proj (gpt-oss: MXFP4 expert blocks), token embedding, lm_head, an RMSNorm vector; KV cache K/V from a fixed ~1024-token forward pass.
  • Two representations per tensor: fp32 (cast to 4-byte float, the width BPC/GPULZ target) and native (stored runtime dtype = bf16; MXFP4 U8 for gpt-oss MLP). Identical bytes go to every tool; buffers are contiguous slices capped at 8 MB (fp32) / 4 MB (bf16).
  • Tools: book LZSS (12-bit window) & shrink -l; GPULZ (CUDA 12.8, sm_120, uint32 symbol); BPC reference model (128 B line, 32×32-bit); gzip -9; zstd -19.
  • Verification: LZSS/shrink round-trips cmp-checked (182/182), GPULZ self-verify (91/91), 0 anomalies.

3. Five findings

(1) Representation dominates everything

Casting to fp32 roughly doubles every tool's ratio versus native bf16 (zstd weights 2.42 → 1.30; BPC 2.26 → 1.25). It isn't extra information — a fp32 copy of a bf16 value pads 16 low/zero mantissa bits and aligns exponents, manufacturing redundancy. bf16 is close to incompressible for the LZ-family tools. Any LLM-compression number is meaningless without stating its dtype.

(2) GPULZ is the wrong tool for LLM tensors

It expands data in every category (0.96–0.99). Its symbol is a 4-byte uint32 and its window is 32 elements, so it needs four consecutive bytes to repeat exactly within 128 bytes — which almost never happens in high-entropy weight/KV bytes. GPULZ is built for smooth multi-byte scientific arrays (where it hit 11.5× on a float field in our earlier test), not neural-network parameters.

(3) BPC is the best of the three specialized tools on weights

It matches gzip on fp32 (2.26) and leads on the smoothest tensors, because its Delta→Bit-Plane→XOR transform captures the sign/exponent value-locality of FP weights — the homogeneous-array structure it was designed for. On native bf16 it drops to ~1.25, and on KV cache it is beaten by the general compressors.

(4) KV cache is far more compressible than weights — but mainly by general compressors

On fp32, zstd reaches 4.5× on KV vs 2.4× on weights; gzip 4.1× vs 2.3×. The fixed-pattern tools don't capture it: BPC stays ~2.2 (fp32) / 1.23 (native), and LZSS reaches 1.43 on native KV — notably its best native-dtype result, beating BPC there.

(5) Already-quantized (MXFP4) weights are incompressible

gpt-oss's 4-bit packed MLP experts give ratios of 0.89–1.04 for every tool — quantization already removed the redundancy, leaving near-random bytes. Lossless compression on top of aggressive quantization buys nothing.

5. What this means

  • For lossless compression of LLM tensors, a general-purpose entropy+dictionary coder (zstd) is the strongest baseline by a wide margin, especially on KV cache.
  • Among hardware memory compressors, BPC is the most promising for floating-point weights; GPULZ and plain LZSS are poorly matched to NN byte distributions.
  • The biggest lever is representation/precision, not the codec: fp32 is ~2× more compressible than bf16, and 4-bit MXFP4 is already past the point where lossless compression helps. For real LLM memory-bandwidth savings, quantization + light entropy coding on the raw bf16 bytes will beat these LZ/bit-plane schemes.

6. Methodology, fairness & limitations

  • Identical bytes per representation — ratios differ only by algorithm.
  • Native ratio definitions, with caveats. BPC is an analytical model: it computes the compressed bit-length, not a decodable bitstream, and is not round-trip-verified — an optimistic upper bound on a real BPC encoder. GPULZ's ratio includes its per-block metadata; it is a real, self-verified codec. LZSS/shrink/gzip/zstd ratios are actual output bytes (LZSS/shrink cmp-verified).
  • 8 MB slices bound runtime (book-LZSS is O(n·D)); ratios are stable across layers, so slicing does not bias the comparison.
  • gpt-oss KV omitted: keeping gpt-oss MXFP4 quantized for inference on a 16 GB GPU is unsupported by the installed stack (transformers dequantizes the MoE to bf16 → ~40 GB → OOM). Its weights are included; KV is a clean same-dtype (bf16) Qwen+Mistral study.
  • Single prompt / fixed length for KV; zstd -19 / gzip -9; gpt-oss weights are dequantized-bf16 except the MLP experts kept as packed U8.
