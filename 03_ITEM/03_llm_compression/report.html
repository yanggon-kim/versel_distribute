LLM Weight & KV-Cache Compression — Comprehensive Results

Every compression algorithm we have (12) measured on weights and KV cache from every cached model (8), to verify earlier results and give one broad, fair, fully cross-referenced comparison. All ratios are original ÷ compressed (geometric mean, higher = better).

2580 measurements = 215 buffers × 12 codecs  ·  8 models, 124M–20B  ·  weights + KV, native & fp32  ·  round-trip verified  ·  RTX 5080, torch 2.9.1 / cramjam / constriction.
Two questions. (1) Do the earlier 3-model numbers hold up? (2) With the full codec set — adding xz, brotli, bzip2, lz4, snappy alongside the LZSS/GPULZ/BPC/gzip/zstd/ANS we already had — what actually wins on LLM weights vs KV cache, and how does it scale with model size?

Weights are entropy-bound; KV cache is repetition-bound. On weights the byte-LZ codecs (lz4/snappy/LZSS/shrink/GPULZ) sit at ≈1.0× — there is no repetition to find — and every gain comes from entropy/transform coding, bunched in a tight band (bzip2 1.38 ≳ brotli 1.38 ≈ xz 1.37 > zstd 1.28 > gzip 1.25 ≈ rANS 1.27 > BPC 1.24). On KV cache the dictionary stage pays off and window size dominates: xz (largest dictionary) leads at 2.65×, then brotli > zstd > bzip2 > gzip, with even lz4 useful (1.75×) while pure entropy (rANS 1.24×) lags far behind.

Representation effect: fp32 roughly doubles every ratio versus native bf16 (KV/fp32 xz hits 4.87×) — not extra information, but the 16 zero/structured bits a fp32 copy adds to each bf16 value. Quote the dtype with any LLM-compression number.

The KV ranking follows window size almost exactly — xz (64 MB) > brotli (16 MB) > zstd (8 MB) > bzip2 (900 KB) > gzip (32 KB) — because KV gains come from long-range repetition. The memoryless entropy coders (rANS) and the small-window / 4-byte-symbol tools (GPULZ, BPC) can't see that repetition, so they trail on KV but stay competitive on entropy-bound weights.

The strongest-ratio codecs are the slowest: brotli-11 (1.2 MB/s) and xz-9 (3.2 MB/s) spend it all in exhaustive/large-window match search. rANS and BPC are ~100× faster (no search) — which is why, when weights are entropy-bound and an entropy coder already ≈ matches gzip, a fast ANS coder is the pragmatic choice. GPULZ's raw op-count is huge but hidden by GPU parallelism.

8. Deep dive: does an entropy coder on the BPC transform help?

BPC = a DBX transform (Delta → Bit-Plane → XOR) followed by BPC's own static RLE + frequent-pattern code map. A natural question: replace that static coder with a real entropy/dictionary coder on the transformed data — does it beat plain BPC? We reimplemented the DBX transform and compressed it with rANS / zstd / bzip2, vs plain BPC and vs the same coders on the raw bytes. Geomean ratio:

data / reprplain ANSplain BPCraw zstdraw bzip2DBX→ansDBX→zstdDBX→bzip2
weight / fp321.8262.0812.2362.5441.7262.0602.004
weight / native1.2711.2391.2801.3821.1561.2631.224
KV / fp321.7832.0083.7463.8531.6803.9383.310
KV / native1.2441.2092.2822.1471.1352.2991.955
No — pairing the BPC transform with a generic coder does not beat plain BPC. DBX→ANS is the worst option in every row: order-0 ANS is memoryless, so it cannot exploit BPC's main lever — runs of all-zero bit-planes (which BPC's static ZBP-RLE encodes in 3–7 bits). A real RLE-capable coder (zstd/bzip2) on the transform only matches plain BPC on weights, and the transform actually hurts general coders: running bzip2/zstd on the raw bytes beats every DBX variant on weights (the delta+XOR scrambles the clean exponent-byte structure they already exploit).

One genuine exception: DBX→zstd on KV cache (3.94× fp32 / 2.30× native) slightly beats raw zstd — the delta exposes cross-element structure a dictionary coder likes. Takeaway: BPC's bespoke static coder is well-matched to its own transform; otherwise prefer plain bzip2 on raw weights and xz/zstd on raw KV. (Note: on native bf16 the DBX transform is a dtype mismatch — it assumes 32-bit words.)

10. Conclusions

  • Weights → entropy-bound. Dictionary LZ finds nothing (≈1.0×); all gain is entropy/transform coding (~1.25–1.5× on bf16). Best lossless: bzip2 (BWT suits exponent bytes), or ANS when speed matters (same ballpark, ~100× faster).
  • KV cache → repetition-bound. Window size dominates; best lossless: xz/LZMA (2.6–3.3× on modern-model bf16 KV), with the whole dictionary family well ahead of pure entropy.
  • Representation dominates (fp32 ≈ 2× native) and already-quantized data (MXFP4) is near-incompressible — lossless compression and quantization are complementary, not additive.
  • Scaling is flat for bf16 weights ≥0.5 B; KV compressibility is model-dependent and fails on the oldest/smallest model (gpt2).

11. Methodology & limitations

  • Verified: 1720 round-trip-verified codec runs (lzss/shrink/gzip/zstd/bzip2/xz/brotli + rANS); 0 anomalies. BPC is analytic (bit-length only); GPULZ self-verifies and its ratio includes block metadata.
  • gpt-oss-20b KV omitted (MXFP4 dequant OOMs on 16 GB); its weights are included.
  • gpt2 is fp32-native (no bf16) — a genuine outlier; read it apart from the bf16 models.
  • 8 MiB slices per tensor (ratios are size-stable); single fixed prompt for KV.
  • Transform / INT8 experiments (byte-plane split, exponent-delta, INT8 quant) and the related-work reproductions live alongside this study; codec levels are the max-quality settings (gzip-9, zstd-19, bzip2-9, xz-9, brotli-11).
