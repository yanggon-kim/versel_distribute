Compressing LLM Weights & KV Cache

A fair, verified compression-ratio study of memory-oriented compressors — LZSS, GPULZ, BPC — against gzip/zstd baselines and the ANS/tANS/rANS entropy-coding floor, on real weights and KV-cache tensors from gpt-oss-20b, Qwen2.5-3B, and Mistral-7B.

Models: gpt-oss-20b (MXFP4), Qwen2.5-3B, Mistral-7B-Instruct (bf16)  ·  Data: weights + KV cache, in fp32 and native bf16  ·  9 codecs incl. ANS/tANS/rANS entropy coders  ·  Host: RTX 5080, CUDA 12.8, torch 2.9.1, transformers 4.57.1.
The question. How well do GPU/memory compressors built for other data — LZSS (byte dictionary), GPULZ (GPU LZSS on 4-byte symbols), BPC (bit-plane transform) — compress actual LLM tensors, and how do they stack up against everyday gzip/zstd? All ratios are original ÷ compressed (higher = better), reported as the geometric mean.

1. Setup

  • Tensors (layers {0, mid, last}, deterministic): attention q_proj/o_proj, MLP down_proj (gpt-oss: MXFP4 expert blocks), token embedding, lm_head, an RMSNorm vector; KV cache K/V from a fixed ~1024-token forward pass.
  • Two representations per tensor: fp32 (cast to 4-byte float, the width BPC/GPULZ target) and native (stored runtime dtype = bf16; MXFP4 U8 for gpt-oss MLP). Identical bytes go to every tool; buffers are contiguous slices capped at 8 MB (fp32) / 4 MB (bf16).
  • Tools: book LZSS (12-bit window) & shrink -l; GPULZ (CUDA 12.8, sm_120, uint32 symbol); BPC reference model (128 B line, 32×32-bit); gzip -9; zstd -19.
  • Verification: LZSS/shrink round-trips cmp-checked (182/182), GPULZ self-verify (91/91), 0 anomalies.

The pure entropy floor (ANS/tANS/rANS) nearly matches gzip/zstd on native bf16 weights (~1.28) — bf16 weights have almost no LZ-exploitable repetition — but falls far short on KV cache (1.9 vs zstd's 4.5), where the gains are mostly repetition, not entropy.

3. Five findings

(1) Representation dominates everything

Casting to fp32 roughly doubles every tool's ratio versus native bf16 (zstd weights 2.42 → 1.30; BPC 2.26 → 1.25). It isn't extra information — a fp32 copy of a bf16 value pads 16 low/zero mantissa bits and aligns exponents, manufacturing redundancy. bf16 is close to incompressible for the LZ-family tools. Any LLM-compression number is meaningless without stating its dtype.

(2) GPULZ is the wrong tool for LLM tensors

It expands data in every category (0.96–0.99). Its symbol is a 4-byte uint32 and its window is 32 elements, so it needs four consecutive bytes to repeat exactly within 128 bytes — which almost never happens in high-entropy weight/KV bytes. GPULZ is built for smooth multi-byte scientific arrays (where it hit 11.5× on a float field in our earlier test), not neural-network parameters.

(3) BPC is the best of the three specialized tools on weights

It matches gzip on fp32 (2.26) and leads on the smoothest tensors, because its Delta→Bit-Plane→XOR transform captures the sign/exponent value-locality of FP weights — the homogeneous-array structure it was designed for. On native bf16 it drops to ~1.25, and on KV cache it is beaten by the general compressors.

(4) KV cache is far more compressible than weights — but mainly by general compressors

On fp32, zstd reaches 4.5× on KV vs 2.4× on weights; gzip 4.1× vs 2.3×. The fixed-pattern tools don't capture it: BPC stays ~2.2 (fp32) / 1.23 (native), and LZSS reaches 1.43 on native KV — notably its best native-dtype result, beating BPC there.

(5) Already-quantized (MXFP4) weights are incompressible

gpt-oss's 4-bit packed MLP experts give ratios of 0.89–1.04 for every tool — quantization already removed the redundancy, leaving near-random bytes. Lossless compression on top of aggressive quantization buys nothing.

5. Compression granularity

"Granularity" splits into three axes that explain the ratio differences: the independent block (unit compressed/decoded in isolation — the unit of parallelism and random access), the reference window (how far back a match may point — bounds the ratio), and the atomic symbol (byte vs 4-byte word). gzip and zstd carry the largest windows, which is the main reason they win here.

GPULZ2 KB block (independent, 1 per GPU thread-block)128 B (32 elems, intra-block)4-byte uint32
BPC128 B line (independent)none — transform within the line only32-bit word
gzip (DEFLATE)variable DEFLATE block, shares window32 KB sliding window1 byte
zstd (−19)block ≤ 128 KB within a frame~8 MB window (level-dependent)1 byte
ANS / tANS / rANSstreaming (memoryless)none — order-0, no dictionary8- or 16-bit symbol
Only BPC (128 B) and GPULZ (2 KB) have truly independent blocks — that is what lets them run in hardware / on the GPU with random access, but it also caps their reach (BPC has no cross-line window at all). The software coders trade that for large byte-granular windows: gzip 32 KB, zstd up to ~8 MB — directly explaining their higher ratios on LLM tensors. The 4-byte symbol of GPULZ/BPC, meanwhile, only matches at 4-byte alignment, which is hostile to byte-packed bf16. (BPC's line size and zstd's window are configurable; the rest are fixed in these builds.)

Compression cost is asymmetric: nearly all of it is the match search (decompression is a cheap linear pass for every tool). Measured below is single-thread wall-clock on one 8 MiB fp32 buffer; for gzip/zstd it is strongly level-dependent (the ratio tables used the heavy levels gzip −9 / zstd −19).

Heaviest compress: gzip −9, then zstd −19 — the cost is the exhaustive/optimal match search plus entropy coding (the high levels we used for the ratio tables). GPULZ does the most raw operations but parallelizes them on the GPU. BPC is lightest by far — no search at all, which is why it is implementable in hardware (~11-cycle latency, ~68K gates). See the gzip & zstd explainer for why the search dominates.

7. Entropy coders (ANS family): the order-0 floor

Added a pure entropy-coding reference with no dictionary or window — it measures the order-0 entropy floor of the raw bytes. Three variants of the same model: ANS (ideal Shannon order-0 bound), rANS (a real range-ANS coder via constriction, round-trip verified on 201 buffers), and tANS (table-quantized ANS). Each run for 8- and 16-bit symbols and raw vs delta inputs (1092 measurements). ANS/tANS/rANS are the same order-0 model — FSE inside zstd is tANS — so they agree to within 0.14% on average (max 1.19%); the tiny spread is only table precision (ANS ≥ rANS ≈ tANS).

Three findings

  • The entropy floor ≈ gzip/zstd on native bf16 weights. Pure 8-bit ANS hits 1.285, matching zstd (1.297) and gzip (1.273) — bf16 weights have almost no LZ-exploitable repetition; nearly all their compressibility is order-0 entropy.
  • KV cache is mostly repetition, not entropy. The gap is large — ANS ≈ 1.94 / 1.27 vs zstd 4.51 / 2.65 — so the dictionary stage, not the entropy stage, is what makes KV cache compressible.
  • Wider symbols help; the delta predictor hurts. 16-bit order-0 beats 8-bit everywhere (fp32 weights 1.95 → 2.56, even edging byte-based zstd's 2.42). A per-symbol delta makes things worse (8-bit pooled 1.58 → 1.15): unlike smooth scientific data, neighbouring LLM tensor elements aren't close, so differencing raises entropy — which is also why BPC's delta/bit-plane transform gains little here.
Caveat. ANS and tANS are analytic (exact for the order-0 model); rANS is a real, round-trip-verified codec whose measured bytes match the analytic numbers to ~0.1%. 16-bit tANS cost is computed from the 218-quantized histogram (a real encode would need that table).

Byte-plane split is a cheap, large win — layout matters more than the codec. Disaggregating the structured high byte (sign+exponent) from the near-random low byte lifts zstd on weights 1.30 → 1.49× (paper's bit-plane reached 1.34) and on KV cache 2.60 → 3.31× (paper 1.8–1.9, peak 2.69) — reproducing 2503.18869's central claim. Our simple per-element exponent-delta slightly hurt (3.31 → 3.14): the full method's cross-token channel grouping needs KV laid out as [tokens×channels], which our flattened slice doesn't preserve.
Not reproducible here: the hardware memory controller (SystemVerilog/7 nm) — algorithmic ratios only; the nvcomp GPU library — CPU cramjam gives identical ratios; exact cross-token channel grouping — approximated; and neural next-token text compression (2505.06297) — a different domain, not run.

9. What this means

  • For lossless compression of LLM tensors, a general-purpose entropy+dictionary coder (zstd) is the strongest baseline by a wide margin, especially on KV cache.
  • Among hardware memory compressors, BPC is the most promising for floating-point weights; GPULZ and plain LZSS are poorly matched to NN byte distributions.
  • The biggest lever is representation/precision, not the codec: fp32 is ~2× more compressible than bf16, and 4-bit MXFP4 is already past the point where lossless compression helps. For real LLM memory-bandwidth savings, quantization + light entropy coding on the raw bf16 bytes will beat these LZ/bit-plane schemes.
  • The order-0 entropy floor (ANS/tANS/rANS) explains where compressibility comes from: on bf16 weights it already matches gzip/zstd (compressibility is entropy, not repetition), but on KV cache it lags far behind them (compressibility is repetition the LZ stage captures). A cheap entropy coder is thus most of what you need for weights; KV cache wants a dictionary stage too.

10. Methodology, fairness & limitations

  • Identical bytes per representation — ratios differ only by algorithm.
  • Native ratio definitions, with caveats. BPC is an analytical model: it computes the compressed bit-length, not a decodable bitstream, and is not round-trip-verified — an optimistic upper bound on a real BPC encoder. GPULZ's ratio includes its per-block metadata; it is a real, self-verified codec. LZSS/shrink/gzip/zstd ratios are actual output bytes (LZSS/shrink cmp-verified).
  • 8 MB slices bound runtime (book-LZSS is O(n·D)); ratios are stable across layers, so slicing does not bias the comparison.
  • gpt-oss KV omitted: keeping gpt-oss MXFP4 quantized for inference on a 16 GB GPU is unsupported by the installed stack (transformers dequantizes the MoE to bf16 → ~40 GB → OOM). Its weights are included; KV is a clean same-dtype (bf16) Qwen+Mistral study.
  • Single prompt / fixed length for KV; zstd -19 / gzip -9; gpt-oss weights are dequantized-bf16 except the MLP experts kept as packed U8.
