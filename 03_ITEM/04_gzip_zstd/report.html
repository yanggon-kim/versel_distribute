gzip (DEFLATE) & zstd

The two everyday lossless compressors, from concept to pseudocode: both follow the same match-finding → entropy-coding shape, but differ in how hard they search and how tightly they entropy-code — which is exactly why zstd is the stronger, more flexible successor to gzip.

gzip: DEFLATE = LZ77/LZSS + Huffman (RFC 1951, 1996)  ·  zstd: LZ + FSE (tANS) / Huffman + optimal parsing (Facebook, RFC 8878)  ·  Companion to: the LLM weight/KV compression study.
The shared blueprint. Almost every general-purpose lossless compressor is two stages: (1) a dictionary / LZ stage that replaces repeated byte sequences with short (offset, length) back-references (turning data into a stream of literals and matches), then (2) an entropy-coding stage that packs those symbols into close to their information-theoretic minimum number of bits. gzip and zstd are two points on this design space — zstd just searches harder and codes tighter.

1. The two-stage pipeline

raw bytes ──▶ [ LZ match-finder ] ──▶ literals + (offset,length) matches ──▶ [ entropy coder ] ──▶ bitstream find repeats in a "the next 5 bytes already pack each symbol near sliding window appeared 1200 bytes ago" -log2(probability) bits
  • Stage 1 (LZ) removes repetition (a phrase seen before becomes a pointer). This is the same LZ77/LZSS idea covered in the LZSS report; the cost lives in searching the window for the longest match.
  • Stage 2 (entropy) removes statistical bias (frequent symbols get short codes). gzip uses Huffman; zstd uses FSE (a tANS variant) plus Huffman.

2. gzip = DEFLATE

gzip wraps the DEFLATE algorithm (a header/CRC around a DEFLATE stream). DEFLATE = LZ77 with a 32 KB sliding window, matches of length 3–258, then Huffman coding of the result.

Match finding: hash chains + lazy matching

DEFLATE hashes the next 3 bytes to index a chain of earlier positions with the same hash, then walks that chain looking for the longest match. The compression level (1–9) sets how deep it walks — higher levels follow longer chains and use lazy matching (defer a match if starting one byte later is longer), trading speed for ratio. 

function deflate(input):                       # 32 KB window, min match 3, max 258
    head = {}                                     # hash(3 bytes) -> most recent position
    pos = 0
    while pos < len(input):
        h = hash3(input[pos:pos+3])
        cand = head[h]                            # walk the chain of prior same-hash positions
        best_len, best_dist = 0, 0
        depth = chain_limit(level)                # level 1 = shallow, level 9 = deep
        while cand != NONE and depth-- > 0 and pos-cand <= 32768:
            L = match_len(input, cand, pos)       # compare bytes, cap at 258
            if L > best_len: best_len, best_dist = L, pos-cand
            cand = prev[cand]
        if best_len >= 3:
            if lazy and match_at(pos+1) longer: emit_literal(input[pos]); pos += 1; continue
            emit_match(best_len, best_dist); advance pos by best_len
        else:
            emit_literal(input[pos]); pos += 1
        insert pos into hash chain

    # Stage 2: Huffman-code the emitted symbols, per DEFLATE block
    for each block:
        build two Huffman trees: {literals + lengths} and {distances}
        pick block type: stored (incompressible) | fixed Huffman | dynamic Huffman
        write trees (if dynamic) + the coded symbols

DEFLATE emits a sequence of blocks, each independently typed: stored (raw, for incompressible data — this is why DEFLATE never inflates much), fixed Huffman (a canned tree, no tree to transmit), or dynamic Huffman (an optimal per-block tree, sent in the header). The LZ window of 32 KB spans across blocks.

3. zstd = bigger window + FSE + optimal parsing

Zstandard keeps the same two-stage shape but upgrades every part. It is Pareto-better than gzip: faster at low levels and much higher ratio at high levels.

What zstd changes

  • Much larger, configurable window (windowLog) — from ~256 KB up to gigabytes; ~8 MB at level 19 vs gzip's fixed 32 KB. Longer-range repeats become compressible. Data is carried in blocks ≤ 128 KB grouped into a frame.
  • Tiered match finders by level: fast (one hash) → dfast (double hash) → greedy/lazy/lazy2 (hash chains) → btlazy2/btopt/btultra (binary tree + optimal parse). Higher levels search dramatically harder.
  • FSE (Finite State Entropy), a fast tANS implementation, codes the sequences (literal-lengths, match-lengths, offsets). Unlike Huffman's whole-bit codes, tANS approaches the entropy limit with fractional bits per symbol. Literals are Huffman-coded.
  • Repeat-offset codes (a small cache of recent offsets) cheaply encode structured, strided data; and an optional trained dictionary primes the window for many tiny inputs.

The compute-heavy part: optimal parsing (high levels)

Greedy "take the longest match now" is not optimal — a shorter match now can enable a much better one next. High zstd levels instead compute the minimum-cost parse of a window using estimated bit-costs of each candidate, via dynamic programming, then re-estimate costs and repeat. This is where the time at -19 goes. 

function zstd_compress(input, level):
    for each block (≤ 128 KB) in input:
        if level is low:
            seqs = greedy_or_lazy_match(block, window)        # hash / double-hash / chains
        else:                                                # btopt / btultra
            costs[i] = estimated_bits_to_encode(symbol stats) # initial price table
            repeat a few times:
                # DP: cheapest way to reach each position using literal or any match
                for i in 0..len(block):
                    price[i+1] = min(price[i] + lit_cost(block[i]),
                                     min over matches m at i of price[i] + match_cost(m))
                seqs = backtrace(price)                       # optimal literal/match parse
                recompute costs from the chosen seqs          # prices converge
        # Stage 2: entropy-code this block
        huffman_encode(literals)
        FSE_encode(literal_lengths); FSE_encode(match_lengths); FSE_encode(offsets)
        emit block (raw | RLE | compressed, whichever is smaller)

Why zstd usually wins. A bigger window finds longer-range repeats; tANS gets closer to the entropy limit than Huffman's whole-bit codes; optimal parsing picks a better match sequence; and repeat-offsets/dictionaries exploit structure gzip can't. Yet at low levels zstd is also faster than gzip — it dominates the speed/ratio trade-off.

5. Decompression is cheap for both

Both are asymmetric: all the effort is in compression's match search. Decoding is a linear pass — entropy-decode the symbols, then copy literals through and copy matches from the already-decoded output (the window). No search, so decompression speed barely depends on the compression level used. This is why "compress once, decompress many" (software/package distribution, web assets, model checkpoints) suits both.

6. How they did on LLM data

In the LLM weight/KV-cache study, these two general-purpose coders were the strongest of all tools tested — their large windows and entropy stages capture structure that the fixed-pattern memory compressors (BPC, GPULZ) and the small-window byte LZSS miss. zstd −19 reached ~2.4× on fp32 weights and ~4.5× on fp32 KV cache, edging gzip −9 throughout — at the cost of being the most compute-heavy on the compress side (optimal parsing).
