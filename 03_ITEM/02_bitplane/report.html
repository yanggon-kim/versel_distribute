Bit-Plane Compression (BPC)

How a lightweight transform (Delta → Bit-Plane → XOR) plus a tiny run-length / frequent-pattern encoder squeezes 4.1:1 out of integer memory traffic on many-core GPUs — the idea, the code map, and what it actually achieves on real data.

Paper: Kim, Sullivan, Choukse, Erez, "Bit-Plane Compression…", ISCA 2016 (UT Austin & NVIDIA)  ·  Scope: motivation → DBX transform → encoder → results  ·  Measurements: the reference BPC model run on real data (RTX 5080 host).
The core idea. Most compressors just encode. BPC first transforms the data into a shape that is far easier to encode — turning a homogeneously-typed memory block into long runs of zeros and a handful of simple bit patterns — and only then applies a cheap encoder. The transform is the secret; the encoder can stay small enough for hardware.

1. Why memory compression, and why it is hard

Many-core throughput keeps climbing, but off-chip memory bandwidth scales poorly — pin counts and energy, not transistors, are the limit. The memory link becomes the bottleneck. Compressing data on the link raises effective bandwidth without adding pins. But memory compression is constrained in ways file compression is not:

  • Tiny blocks. A compressor sees a half/full cache line (64–128 B), not a whole file — so large dictionaries (LZ, Huffman tables) don't fit and don't pay off.
  • Low latency & random access. It sits on the critical path of every memory access, so heavy entropy coders are out.
  • Coarse transfer granularity. DRAM bursts can erase fine-grained bit savings unless the system packs compressed blocks densely.

As a result, prior memory compressors reach only ~2:1 for integers and rarely beat 1.5:1 for floats. BPC's goal is to do much better while staying cheap.

2. The key insight: homogeneous arrays have value locality

GPGPU programs follow an SPMD model and keep almost all data in large arrays. A survey of Rodinia / Parboil / Lonestar benchmarks shows global memory is overwhelmingly homogeneously-typed arrays (primitive int/float, or homogeneous structs) with negligible scalar data. Homogeneous neighbors have value locality — adjacent words are numerically close. A transform that exposes this locality, then a transform that turns it into zeros, makes the data trivially compressible.

3. The DBX transform (Delta → Bit-Plane → XOR)

BPC views a 128 B block as 32 × 32-bit words and applies three cheap, reversible steps before any encoding:

  1. Delta — subtract neighboring words. The 32 words become a base (the first word) plus 31 deltas (each 33-bit, one extra bit for the borrow). Value locality ⇒ small deltas ⇒ reduced range and entropy.
  2. Bit-Plane (BP) — transpose the delta words. A bit-plane is the set of bits at the same position across all words. This re-orients the data so value locality shows up as bit-level locality (whole planes become uniform).
  3. XOR neighboring bit-planes: DBXₖ = DBPₖ ⊕ DBPₖ₊₁. Adjacent planes are nearly identical (small deltas ⇒ upper planes are all sign bits), so the XOR collapses them into many all-zero symbols, especially the upper planes.
32 words (128 B) Delta Bit-Plane (transpose) XOR neighbors w0 w1 w2 ... w31 -> base, d0..d30 -> plane[0..32] across words -> DBX (mostly 0) (small values) (rows = bit positions) upper planes -> 0 0 0 0 ... Result: 33 symbols = 1 base bit-plane + 32 DBX planes, each a 31-bit bit-plane symbol, dominated by ALL-ZERO planes + a few simple patterns (single 1, two 1s, all 1s).

DBX raises the inherent compressibility of the block by ~87% (INT), 23% (FP), 20% (mixed) on average. It maps directly onto the reference code's compressLine(): 

// 31 deltas between consecutive 32-bit words (base = word[0])
for (i = 1; i < 32; i++)
    deltas[i-1] = (INT64) s_dword[i] - (INT64) s_dword[i-1];

// build each bit-plane by transposing, then XOR with the previous plane
for (j = 63; j >= 0; j--) {
    buf = 0;
    for (i = 30; i >= 0; i--) { buf <<= 1; buf |= (deltas[i] >> j) & 1; }   // transpose
    DBP[j] = buf;
    DBX[j] = buf ^ prevDBP;          // XOR neighboring bit-planes
    prevDBP = buf;
}
blkLength  = encodeFirst(word[0]);   // base word, encoded separately
blkLength += encodeDeltas(DBP, DBX); // the 33 bit-plane symbols

4. The BPC encoder: ZRL + frequent patterns

Because DBX manufactures long zero runs and simple patterns, the encoder can be tiny — no 2⁷ entry Huffman table. Zero bit-plane runs go through Zero Bit-Plane Run-Length Encoding (ZBP-RLE); non-zero planes match a handful of fixed frequent-pattern codes. This is the code map (paper Fig 3b), implemented one-for-one in encodeDeltas():

The base word is encoded separately by encodeFirst(): 0 → 3-bit; fits signed 4/8/16-bit → 3+4 / 3+8 / 3+16; otherwise 1+32-bit. Note the worst case is just one flag bit of overhead, so BPC never inflates badly.

5. What it achieves

Compression ratio vs prior memory compressors (FPC / BDI / C-pack / BPC), from the paper:

Data classFPCBDIC-packBPC
Integer (homogeneous)1.62.32.24.1
Floating point1.21.51.41.9
Mixed1.51.72.12.5
  • BPC never increases average block size; FPC/BDI/C-pack each enlarge some workloads.
  • It beats even the entropic limit of the original symbols in 6/10 integer benchmarks — because the DBX transform creates the long zero runs ZRL exploits.
  • Hardware: Verilog at 40 nm / 800 MHz ≈ 68K NAND2 gates, 11-cycle latency — comparable to C-pack (~40K gates).
  • System: integrated with an HMC-like packetized link and a Packed Packets protocol, BPC reaches 90% / 95% of full-bandwidth performance using 2.0× / 1.9× less bandwidth on demanding benchmarks.

6. Measured here, on the reference model

The repository ships a header-only model of BPC that computes the compressed bit-length per 128 B line (it measures the ratio; it does not emit/decode a real bitstream). Built and run on real inputs:

InputNatureBPC ratio
QCLOUDf48 (float32 scientific field)homogeneous float array, value locality11.65:1
kennedy.xlsheterogeneous binary1.03:1
alice29.txtEnglish text (not a numeric array)0.97:1
Reading these numbers. BPC shines exactly where its model holds — a homogeneous numeric array with value locality (the float field hits 11.65:1, well above the paper's averages because that data is unusually smooth). On heterogeneous binary it neither helps nor hurts (1.03:1), and on text — which is not a same-type numeric array — the delta+bit-plane transform finds nothing to exploit and it slightly expands (0.97:1), bounded by the one-flag-bit overhead. Caveat: these are analytical sizes from the size-only model, not a verified round-trip bitstream.

7. Takeaway

BPC's lesson is transform before you encode. By spending three cheap, hardware-friendly steps (Delta, Bit-Plane, XOR) to convert homogeneously-typed memory blocks into long zero runs and simple patterns, a tiny RLE + frequent-pattern encoder reaches compression ratios (4.1:1 integer, 1.9:1 float) that heavier general-purpose encoders cannot — at low latency and modest area, directly attacking the GPU memory-bandwidth wall. Its weakness is the flip side of its strength: data that isn't a homogeneous numeric array (text, already-random bytes) gives it nothing to transform.
